# 微信自动客服实时响应与 token 成本优化架构方案（2026-05-16）

## 1. 背景

2026-05-16 实盘测试中，客户发送“我预算10万左右，主要上下班通勤，有什么推荐？”后，系统约 6-7 分钟才回复。审计记录显示，消息在 `01:59:51` 被捕获，实际回复在 `02:06:32` 发出。

这说明问题不是单条配置超时太长，而是实时客服主链路把重型 LLM 综合回复放在发送前同步等待。只要 DeepSeek、网络、证据包或 JSON 输出变慢，整个监听循环都会停住。

## 2. 根因

### 2.1 主循环阻塞

`run_customer_service_listener.py` 每一轮用 `subprocess.run(...)` 执行 `listen_and_reply.py --once`。父监听器没有整轮硬超时，子流程卡住时不会继续扫描新消息。

### 2.2 LLM 合成在发送前同步执行

`listen_and_reply.py` 在发送微信消息前调用 `maybe_synthesize_reply(...)`。这让 LLM 响应时间直接变成客户等待时间。

### 2.3 证据包过重

当前合成证据包会包含：

- 当前消息。
- 当前批次。
- 近期聊天历史。
- 结构化商品、FAQ、政策。
- RAG 命中。
- 商品候选。
- 客户画像。
- 平台安全规则。
- 旧回复、旧 LLM advisory、数据采集结果。

在实盘日志中，一次普通预算推荐的 prompt 估算约 13k tokens，真实总用量约 14k tokens。这种规模适合离线诊断，不适合每条客户消息实时等待。

### 2.4 质量目标和实时目标混在一起

旧方案把“拟人化”绑定到“每条消息必须同步 LLM 合成”。实际更合理的拆法是：

- 拟人化风格来自已批准真实客服经验、模板变量、话术库和轻量改写。
- 深度 LLM 负责后台总结、复盘、候选知识和话术持续优化。
- 只有少数确实需要语义综合的消息进入前台轻量 LLM。

## 3. 新架构目标

### 3.1 在线链路

在线链路只做客户实时可感知的事情：

```text
微信消息捕获
  -> 去重与批次合并
  -> 快速意图/风险识别
  -> 商品库/正式知识/RAG 轻量检索
  -> 回复路由
  -> 安全 guard
  -> 发送或请示话术
  -> 记录审计并投递后台任务
```

在线链路必须满足：

- 有整轮耗时预算。
- 有每个外部调用的 timeout。
- 有可发送兜底话术。
- 允许跳过 LLM，但不允许跳过安全 guard。
- 不做深度知识生成、RAG 解释、候选知识晋升。

### 3.2 后台链路

后台链路负责慢任务：

```text
实时审计事件
  -> work queue
  -> LLM 复盘/经验抽取/质量评估
  -> RAG 经验候选
  -> 人工审核
  -> 正式知识或经验库更新
```

后台链路允许慢，但必须：

- 不影响实时回复。
- 有任务状态、重试、失败记录。
- 所有 LLM 结果进入候选或审计，不直接越权写主数据。

## 4. 回复路由分层

### 4.1 L0：无需 LLM

适用：

- 问候。
- 简单确认。
- 留资收集。
- 已明确商品事实。
- 高风险边界请示。
- 明显无关/测试边界。

处理方式：

- 使用规则、商品库、正式知识、已批准模板直接回复。
- 使用客户上下文做少量变量填充。
- 经过安全 guard 后发送。

目标 token：前台 0。

### 4.2 L1：轻量组合回复

适用：

- 预算推荐。
- 用途推荐。
- 简单对比。
- 到店/异地看车流程。
- 客户问法有口语化但意图清楚。

处理方式：

- 程序先检索 Top 2-3 商品。
- 程序先选 Top 1-2 经验片段。
- 本地模板组织回复。
- 如果模板质量不足，再走小包 Flash。

目标 token：0 或 600-1800。

### 4.3 L2：前台轻量 LLM

适用：

- 多轮上下文指代。
- 客户需求较复杂但风险不高。
- 本地模板会显得生硬。
- 需要把商品事实和真实客服风格自然融合。

处理方式：

- 小证据包 Flash。
- 严格限制历史、商品、RAG 数量。
- 一次调用，不做多轮重试。
- 超时直接回退 L1。

目标 token：1500-3000。

### 4.4 L3：后台深度 LLM

适用：

- 复盘回复质量。
- 总结客户画像。
- 生成 RAG 经验。
- 解释 RAG 经验价值。
- 候选知识抽取。
- 复杂风险案例分析。

处理方式：

- 后台 worker 处理。
- 可以使用 Pro。
- 结果进入候选、经验池或审计。

目标：不计入前台耗时预算。

## 5. 核心改造点

### 5.1 主监听 watchdog

每轮 `listen_and_reply.py --once` 必须有硬超时。超过阈值后：

- 终止本轮子进程。
- 写入 `watchdog_timeout` 审计。
- runtime status 显示异常原因。
- 下一轮继续监听。

建议默认：

- 普通生产：20-30 秒。
- 调试：60 秒。
- 禁止无限等待。

### 5.2 LLM 从默认同步改为条件同步

`llm_reply_synthesis` 不再是每条消息默认必经。新增路由器：

```text
ReplyRuntimeRouter
  -> decide_foreground_level()
  -> build_runtime_budget()
  -> choose_reply_builder()
```

路由输出：

- `level`: L0/L1/L2/L3
- `foreground_llm_allowed`: true/false
- `max_prompt_tokens`
- `max_latency_seconds`
- `fallback_reply`
- `background_jobs`

### 5.3 证据包瘦身

前台证据包必须使用 realtime profile：

- 历史消息：最多 4-8 条。
- 商品候选：最多 2-3 个。
- RAG 命中：最多 1-2 条。
- 每条 RAG 文本：120-220 字。
- 平台规则：只传命中的规则摘要，不传长规则全文。
- 客户画像：只传对当前回复有用的字段。

### 5.4 商品事实前置确定

商品推荐不应让 LLM 在大包里“自己挑车”。程序先根据预算、用途、关键词、库存状态、价格区间筛选候选，再把候选给回复层。

商品事实规则：

- 商品库是唯一事实源。
- LLM 不得新增价格、库存、年份、公里数、车况结论。
- RAG 只能给表达风格，不得覆盖商品库事实。

### 5.5 高风险问题前台不等模型

涉及底价、事故水泡火烧保证、赔付、合同、定金、金融审批、发票、过户强承诺时：

- 前台直接发请示/核实话术。
- 对内创建 operator alert。
- 后台可异步总结风险点给人工。

这样既安全，又避免客户等模型判断风险。

## 6. 新旧链路对比

| 项目 | 当前 | 目标 |
|---|---|---|
| 主循环 | 子进程无限等待 | 整轮 watchdog |
| LLM 合成 | 默认同步参与 | 按路由条件参与 |
| 证据包 | 大包综合 | 前台小包，后台大包 |
| 商品推荐 | LLM 看多商品后生成 | 程序先筛，LLM 只表达 |
| 高风险问题 | 可能同步 Pro 判断 | 前台固定请示，后台分析 |
| token 成本 | 常见 10k-20k+ | 常规 0-3k |
| 卡死影响 | 阻塞整个监听 | 单轮失败后恢复 |

## 7. 兼容策略

- 保留现有 `llm_reply_synthesis` 模块，但它不再默认位于所有消息的主路径。
- 保留审计字段 `llm_reply_synthesis`，新增 `runtime_route` 和 `token_budget`。
- 旧配置继续可读；新配置未开启时使用保守默认策略。
- 可以按租户灰度启用。

## 8. 不做事项

- 不删除 RAG 经验层。
- 不删除 LLM 综合回复能力。
- 不让 RAG 或聊天经验反向改商品库。
- 不把所有回复退化成关键词模板。
- 不把高风险问题交给模型自由发挥。
