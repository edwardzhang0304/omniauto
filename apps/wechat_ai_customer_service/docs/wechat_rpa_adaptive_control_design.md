# 微信 RPA 自适应操控方案说明

更新时间：2026-06-17

本文说明 OmniAuto 现有微信聊天操控模块的设计思路，供后续开发者维护、扩展和迁移到新动作时参考。

## 核心定位

当前微信操控方案不是传统意义上的固定坐标脚本。

它的核心思路是：

```text
先识别当前微信状态
再确认目标对象
优先使用 OCR / UIA / 可验证控件信息
必要时才使用几何坐标作为动作落点
每个高风险动作前后都做校验
确认不了就停止，而不是继续盲点
```

因此，它更适合做跨分辨率、跨窗口尺寸、跨 DPI 的适配。分辨率变化时，通常调整的是截图区域、窗口几何、OCR 阈值、UIA 控件选择和坐标兜底范围，而不是重写整条流程。

## 为什么不采用纯固定坐标

纯固定坐标的问题是，它默认微信窗口内部布局永远一致。例如：

```text
点击窗口内 x=354, y=64
```

或者：

```text
根据窗口宽度推算左侧栏分割线，再固定向左偏移若干像素
```

这类方法在单台测试机上可能稳定，但换电脑后容易因为以下因素失效：

- Windows 显示缩放不同。
- 屏幕分辨率和可用工作区不同。
- 微信窗口外框尺寸相同，但客户区尺寸不同。
- 微信版本、皮肤、账号状态、搜索状态导致内部布局变化。
- OCR 截图坐标和鼠标点击坐标存在映射差异。
- 当前页面可能不是聊天主界面，而是登录、白屏、安全提示或辅助窗口。

所以本项目不把固定坐标作为主要判断依据。坐标只应该是动作执行层的最后一步，而不是业务判断依据。

## 分层思路

### 1. 状态识别层

在任何点击、输入、发送之前，先确认微信窗口是否处于可操作状态。

典型检查包括：

- 是否找到 Windows 微信主窗口。
- 窗口是否可见、未最小化、尺寸是否足够。
- 截图是否为空白渲染。
- OCR 是否读到有效内容。
- 是否处于快速登录、二维码登录、重新登录、安全验证等阻断页面。
- 当前窗口是否只是微信辅助壳，而不是真正聊天界面。

如果状态不满足要求，流程应该停止并报告原因。

### 2. OCR 语义识别层

聊天模块通过 OCR 识别微信界面中的可见文本，而不是根据固定行号判断。

主要用途包括：

- 从左侧栏识别会话名、预览消息、时间、未读红点。
- 从聊天区识别消息气泡。
- 区分输入区草稿和真实聊天消息。
- 判断当前聊天标题是否匹配目标会话。
- 识别白屏、登录提示、阻断提示。

OCR 结果是证据，不是最终动作。动作前还需要结合区域、置信度、窗口几何和目标会话校验。

### 3. UIA 控件层

发送消息时，优先使用 Windows UI Automation。

也就是直接查找微信窗口里的输入框控件和“发送”按钮控件，然后通过控件能力写入和触发发送。

这比坐标点击更适合跨分辨率，因为它依赖的是控件树和控件位置，而不是固定屏幕像素。

如果 UIA 不可用，才进入受保护的坐标点击兜底。

### 4. 几何坐标兜底层

几何坐标不是完全不用，而是只在可控范围内作为兜底。

例如：

- 根据窗口宽高估算输入区安全区域。
- 根据左侧栏分割线估算会话行点击区域。
- 根据底部工具栏估算发送按钮区域。
- 在安全区域内生成多个候选点，避免反复机械点击同一像素。

这些坐标必须满足两个条件：

- 坐标不能越过安全区域，例如不能落到会话列表或聊天正文区。
- 动作前后必须有校验，例如目标标题确认、输入区内容确认、发送后窗口状态确认。

### 5. 目标确认层

发送客户可见内容前，必须确认当前会话就是目标会话。

强确认来源是当前聊天窗口标题 OCR 命中目标名称。左侧栏预览、聊天正文、历史消息里出现目标名称，都不能单独授权发送，因为它们可能来自别的会话内容。

如果目标无法强确认，流程应该阻断，而不是继续输入或发送。

### 6. 动作安全层

RPA 动作必须低扰动、可解释、可停止。

现有方案包含这些原则：

- 全局 RPA 锁，避免多个流程同时操作同一个微信窗口。
- 发送频率限制，避免短时间密集发送。
- 点击和输入加入随机停顿，避免机械化节奏。
- 鼠标动作和键盘动作串行，不制造人类不可能的同步反应。
- 搜索、切会话、回滚历史等高风险路径限次执行。
- 发现白屏、登录、安全提示、窗口失焦时立即停止。
- 失败后让调度器低频重试，而不是在当前页面连续盲点。

## 当前聊天模块的执行模型

### 会话列表读取

会话列表通过截图 + OCR 获取。

系统会从 OCR 项里筛选可能的会话名，并结合位置、行间距、时间文本、预览文本、未读红点等信息生成会话候选。

这不是固定读取第几行，而是根据当前截图重新识别可见会话。

### 聊天内容读取

聊天内容通过 OCR 获取可见气泡文本。

系统会过滤标题栏、左侧栏、底部输入区和噪声文本，再根据位置判断消息大致来自自己还是对方。

对于历史回补，优先采用锚点驱动的小步上翻：找到目标锚点就停止，达到步数或时间上限也停止。

### 切换会话

切换会话前，系统先判断当前微信界面是否可操作。

然后优先从 OCR 识别出的会话候选中找到目标，再点击候选会话行。点击后不会默认认为成功，而是重新 OCR 验证当前聊天标题。

搜索切换属于高风险路径，默认更保守，避免无边界搜索和盲按回车。

### 发送消息

发送前必须进行强目标确认。

发送优先走 UIA 控件：

```text
找到输入框控件
设置文本
找到发送按钮控件
触发发送
```

如果 UIA 不可用，才走受保护的点击输入方案：

```text
计算输入区安全点
点击输入区
写入文本
确认输入区可见内容
触发发送
发送后检查窗口状态
```

发送链路不会只因为“点过发送按钮”就认为成功，还会结合发送结果、消息复核、窗口状态和守卫记录判断。

## 和固定坐标方案的差异

| 维度 | 自适应操控方案 | 固定坐标方案 |
| --- | --- | --- |
| 主要依据 | OCR、UIA、窗口状态、目标确认 | 像素位置或固定几何公式 |
| 换分辨率 | 调整适配层即可继续演进 | 容易整体偏移 |
| 点错风险 | 点击前后有校验，失败会停 | 容易盲点、重复点 |
| 可观测性 | 保存截图、OCR、guard、结果状态 | 通常只有成功/失败 |
| 风控表现 | 低频、限次、随机节奏、可阻断 | 容易机械重复 |
| 维护方式 | 优化识别、校验、兜底策略 | 持续调坐标 |

## 对新 RPA 动作的开发要求

后续新增微信动作时，应复用这套思路：

1. 先做状态探测，不在未知页面上点击。
2. 用 OCR/UIA 寻找目标，不把固定坐标当成唯一依据。
3. 坐标只作为兜底，且必须限定在安全区域。
4. 每个关键动作前后都保留证据。
5. 目标不确认就停止。
6. 高风险动作限次、低频、随机节奏。
7. 失败时返回结构化原因，不继续盲目重试。
8. 报告中记录窗口尺寸、截图尺寸、DPI、OCR 项、候选目标和最终点位，便于跨机器排查。

## 加好友模块应如何对齐

加好友模块也应该按同样思路改造：

- 点击“+”前，先确认当前确实是可操作的微信主界面。
- 不只用窗口宽度推算“+”按钮，应结合 OCR/视觉/候选区域确认。
- 点击后必须确认弹出菜单真的出现。
- 点击“添加朋友”前，应识别菜单项或至少确认菜单区域内有对应文本。
- 搜索手机号前，应确认进入了添加朋友页面。
- 输入手机号后，应 OCR 确认搜索框内容或搜索结果状态。
- 点击“添加到通讯录”应基于 OCR 目标项，而不是固定区域。
- 遇到登录、安全验证、操作频繁、账号异常等状态必须停止。
- 所有实机报告必须记录设备画像和窗口画像。

这样加好友能力才能从“某台机器可跑通”升级成“多机器可适配、可诊断、可安全停止”的工程能力。
