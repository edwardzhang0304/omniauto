# Brain First 风险、灰度与未决事项

## 1. 风险总览

| 风险 | 等级 | 原因 | 控制方式 |
| --- | --- | --- | --- |
| LLM 编造事实 | 高 | Brain 权限上移后，模型可能过度发挥 | facts_claimed + guard + 商品库/正式知识回链 |
| 回复变慢 | 中 | Brain 默认参与所有正常对话 | 并发 LLM、证据裁剪、Prompt 缓存、分项耗时观测 |
| 旧 route 行为误伤 | 中 | route 降级可能影响已通过场景 | shadow 对比、fallback、现有测试全跑 |
| 多会话错发 | 高 | Brain 并发但 RPA 发送串行 | ready queue 绑定 target/session，发送前复核 |
| 闲聊过度 | 中 | 允许无害陪聊后可能偏离业务 | soft redirect 策略和长度限制 |
| 商品匹配错误 | 高 | 模糊匹配可能把 A 车识别成 B 车 | LLM 匹配 + 商品库 aliases + confidence + guard |
| Guard 过严导致不回复 | 中 | facts_claimed 要求过高 | 分阶段启用，先 shadow 观察 |
| Guard 过松导致越界 | 高 | 常识层与事实边界混淆 | authority contract 不可配置关闭 |
| 最终润色改坏事实 | 高 | 润色模型可能改价格或政策 | 润色前后 fact diff 复核 |
| AI经验池污染 | 高 | 历史聊天含过期事实 | AI经验池只进 style/audit，不进 content basis |

## 2. 灰度策略

### 2.1 阶段 A：Shadow

配置：

```json
{
  "customer_service_brain": {
    "enabled": true,
    "mode": "shadow"
  }
}
```

行为：

- 旧链路继续出回复。
- Brain 同步生成计划但不发送。
- 对比两者质量、事实来源、耗时。

通过标准：

- Brain 在 90% 以上正常业务场景中给出更相关或不低于旧链路的计划。
- Brain 无事实编造。
- Guard 能拦截越权计划。

### 2.2 阶段 B：Hybrid Shadow

行为：

- 低风险问候、闲聊、明确商品库命中报价、明确推荐进入 Brain。
- 高风险和不确定场景仍旧链路 fallback。
- 记录差异。

通过标准：

- File Transfer 实盘无错发、无白屏、无掉线。
- 回复质量明显优于旧 route。

### 2.3 阶段 C：单会话 Brain First

行为：

- chejin 单会话正常业务默认 Brain。
- 旧链路只用于错误 fallback。

通过标准：

- 20 条真实改写问题通过。
- 没有短句漏回。
- 没有商品事实错误。

### 2.4 阶段 D：双会话 Brain First

行为：

- 许聪 + 新数据测试 双会话交替。
- Brain 可并发，RPA 串行发送。

通过标准：

- 不错发。
- 不串上下文。
- 不因会话切换导致白屏。

### 2.5 阶段 E：默认启用

前置条件：

- 上述阶段全部通过。
- 关键测试纳入回归。
- 有明确回滚开关。

## 3. 回滚方案

任何阶段可回滚：

```json
{
  "customer_service_brain": {
    "enabled": false,
    "mode": "off"
  }
}
```

回滚后：

- 旧链路恢复。
- Brain 日志保留。
- 不删除新增数据合同。
- 不清理审计记录。

触发立即回滚：

- 多会话错发。
- 商品事实编造。
- 正式政策越界承诺。
- 最终润色跳过或改坏事实。
- 实盘触发白屏或掉线。
- Guard 异常导致大量漏回。

## 4. 监控指标

必须监控：

- Brain 启用率。
- Brain 采纳率。
- Brain fallback 率。
- Guard 拦截率。
- final polish 失败率。
- 平均总耗时。
- P90 / P95 总耗时。
- ready 到发送耗时。
- 多会话错位复核失败次数。
- 商品事实 guard 拦截次数。
- AI经验池来源被排除次数。

## 5. 需要保留的旧能力

不要删除：

- 旧 `llm_reply_synthesis` fallback。
- 旧 route 的风险判断。
- RPA 安全护栏。
- 云端授权和配置校验。
- AI经验池治理。
- 记录员模块。
- 飞书/ServerChan 转人工接口。

这些能力是安全网，不是本次重构的敌人。需要调整的是“正常客服回复的决策权归属”。

## 6. 未决事项

### 6.1 是否引入 LLM Verifier

当前 guard 可以先做本地规则校验。后续如果答非所问仍多，可加低成本 verifier：

```text
输入：用户问题 + BrainPlan + 最终回复
输出：是否回答当前问题、是否偏题、是否应追问
```

建议：

- 第一阶段先不引入，避免延迟增加。
- 若实盘仍有明显答非所问，再加。

### 6.2 Brain 和 final polish 是否合并

当前不合并。

原因：

- 用户明确要求所有回复都润色。
- Brain 负责思考和策略。
- final polish 负责客户可见表达。
- 分层更容易审计和防止事实被改坏。

### 6.3 本地固定模板是否彻底删除

不彻底删除。

保留范围：

- 系统异常。
- 微信掉线。
- 白屏。
- 安全护栏失败。
- 转人工提示。

正常销售、问候、报价、推荐、闲聊不走固定模板。

### 6.4 AI经验池是否完全不进入 Prompt

不是完全不进入。

允许进入：

- 脱敏后的风格摘要。
- 常见问法类型。
- 历史异议处理方式的抽象策略。

禁止进入：

- 历史价格。
- 历史库存。
- 历史车型事实。
- 历史客户个人信息。
- 未确认政策。

## 7. 商用前额外建议

在 Brain First 验收后，正式商用前还建议：

- 建立真实客户问题回归集。
- 每周从未命中/转人工/客户质疑中自动抽样复盘。
- 给商品库 alias 维护后台增加“用户错别字/音译建议”。
- 给 final polish 增加事实 diff 检查。
- 给多会话错发风险增加独立红线报警。
- 给 LLM 上游延迟和失败建立供应商对比看板。

## 8. 本次文档确认后的下一步

建议下一轮代码落地按以下顺序：

1. 新增 Brain contract 和 shadow runner。
2. 在 `listen_and_reply.py` 接入 shadow，不改变现有回复。
3. 做离线对比测试。
4. 启用 hybrid_shadow，低风险场景采纳 Brain。
5. 降级 realtime local reply。
6. 强化 guard。
7. 单会话实盘。
8. 双会话实盘。
