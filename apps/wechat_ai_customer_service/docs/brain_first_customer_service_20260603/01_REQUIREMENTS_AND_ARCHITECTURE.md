# Brain First 客服大脑需求与架构

## 客户可见回复所有权硬基线

- 所有客户可见回复必须由 `customer_service_brain` 发出：只能是首个有效 BrainPlan、Brain repair 后的 BrainPlan，或 Brain 自己生成的硬边界/拒绝/转人工类说明。
- Guard、质量门、语义审稿、RAG、实时路由、本地模板、旧合成器、最终润色和任何兜底模块都不能生成、替换、拼接客户可见回复；它们只能提供证据、风险、审稿意见、返修指令或轻量表达校验。
- Brain 不可用、超时、不可采纳或返修失败时，不允许本地 safe fallback 代替 Brain 发客户可见话术；必须阻断发送、记录审计，并触发内部人工/告警接口。
- 后续所有客服相关开发文档必须引用 [customer_visible_reply_ownership_baseline.md](../customer_visible_reply_ownership_baseline.md)。

## 1. 一句话目标

把微信智能客服的正常业务回复从“规则先定答案，LLM 后置润色”升级为“LLM 先理解客户并制定回复策略，商品库和正式知识库提供权威事实，guard 负责校验边界，最终润色负责真人化表达”。

## 2. 为什么必须从根上改

当前系统经过多轮优化后，RPA 稳定性、多会话调度、最终润色、商品库优先、AI经验池治理等基础能力都已经具备。但近期手测暴露的问题不是单点 bug，而是决策架构的问题：

- 客户问“推荐一辆”，系统仍可能机械推荐两辆。
- 客户质疑后，系统还会重复旧推荐，而不是理解质疑并调整策略。
- 客户问“塞纳多少钱”，系统没有充分使用 LLM 的模糊理解能力去匹配“赛那”。
- 客户问无关闲聊时，系统容易生硬转业务，缺少自然陪聊和柔性引导。
- 短问候被认为低信息而跳过，说明本地规则对自然会话覆盖不完整。
- RPA OCR 把用户名混入正文后，LLM 会把身份标签当成客户内容。
- 本地 route 越补越多，但仍无法覆盖真实客户的长尾表达。

这些问题说明：客服的“理解和思考”不能继续由本地规则硬分支承担。

## 3. 当前真实链路诊断

当前主链路大致为：

```text
RPA捕获消息
  -> decide_reply / intent_assist
  -> maybe_build_rag_reply
  -> maybe_apply_llm_reply
  -> decide_realtime_reply_route
  -> maybe_build_realtime_reply
  -> maybe_synthesize_reply
  -> style adapter / final_visible_llm_polish
  -> RPA发送
```

关键问题：

1. RAG 可先生成回复。
2. 配置 `rag_response.skip_llm_after_apply` 默认为可跳过 LLM。
3. `realtime_reply_router` 可直接生成本地客服回复。
4. 多个 route 明确设置 `foreground_llm_allowed = False`。
5. `llm_reply_synthesis` 虽然具备综合能力，但在链路上太靠后。
6. 最终润色只能改变语气，不能改变已经错误的业务策略。

因此，现在实际是：

```text
本地规则决定说什么
LLM 在允许时补充或润色怎么说
```

目标应改为：

```text
LLM 决定如何理解和怎么回复
规则只提供权威证据、安全边界和发送约束
```

补充权限原则：

- Brain First 开启后，所有客户可见回复的最终决定权和可见文本来源只属于 LLM 客服大脑。
- Brain 的“最终决定权”指客户回复的理解、策略、取舍、推荐顺序、回应质疑、闲聊边界和表达组织；商品事实、正式政策、安全边界仍是不可绕过的硬约束。
- 结构化路由、RAG、历史聊天、AI经验池、实时规则和旧合成器只能提供证据、候选、风险提示或审稿意见。
- 如果结构化/质量门发现 Brain 草稿有问题，应把问题作为返修意见交给 Brain 重新思考；不能由本地模板直接替换 Brain 的业务回答。
- 如果 Brain 不可用、超时或返修后仍不可采纳，只允许阻断客户可见发送并进入内部人工/告警接口；任何本地安全兜底、旧结构化业务模板或系统提示都不能代替 Brain 给客户发话。
- 最终润色层只做客户可见表达校验和轻量自然化，不改变 Brain 的事实来源、回复策略和风险判断。

### 3.1 Brain 主控不可降级原则

后续所有客服回复质量优化都必须遵守这条方向：能交给 Brain 的都交给 Brain。

- 结构化模块不是第二套客服大脑，只能承担证据检索、事实归属、风险识别、候选提示、审稿反馈和发送约束。
- guard、质量门、语义审稿、最终润色如果发现问题，应生成“返修意见”交给 Brain 重新思考，而不是用本地规则、固定模板或旧合成器直接改写客户回复。
- 最终决定“客户到底问了什么、应该先回答什么、推荐哪台、如何回应质疑、是否陪聊、是否转人工”的主体只能是 Brain。
- 润色层被定位为弱后置层：可以检查不自然、过长、省略号、半句、明显风险表达，并做轻量自然化；不能改 Brain 的事实依据、业务策略、推荐对象、风险判断和会话意图。
- 优化代码时要优先做全局通用能力：更好的 Brain 输入、更清晰的权威证据、更智能的返修反馈、更稳的多会话隔离、更可靠的审计。禁止把一次坏回复修成一条车型、关键词、预算或话术硬编码。
- 只有当问题本质是权威数据缺失、商品库错误、正式知识缺失、安全边界缺失、OCR/RPA元数据污染或多会话绑定错误时，才允许改结构化数据、规则或底层校验。

### 3.2 纠偏决策准则

每次修复客服回复质量问题前，先判断问题归因：

| 问题类型 | 优先修复位置 | 不应采取的做法 |
| --- | --- | --- |
| Brain 没理解客户意图、质疑、上下文或闲聊 | Brain prompt、Brain 输入摘要、会话上下文、返修反馈 | 新增关键词模板直接给答案 |
| Brain 用错证据或缺证据 | evidence pack、商品库/正式知识检索、证据压缩策略 | 把历史聊天或AI经验池提权为事实 |
| Brain 事实越权或承诺过度 | guard 反馈、Brain repair 提示、正式知识边界 | 让 guard 直接生成替代回复 |
| 回复不自然、过长、半句、省略号 | Brain 输出合同、最终润色弱校验、分段策略 | 让润色层改推荐策略或事实 |
| OCR/RPA 把用户名、群名、标题混入正文 | 捕获清洗、metadata envelope、测试用例 | 在 Brain 里硬猜哪些词是用户名 |
| 多会话串话、错发、上下文污染 | 会话ID绑定、任务队列、发送前复核 | 用更严格话术模板掩盖错位 |

如果修复方案不能落到上表的“优先修复位置”，应先停下来重新评估，避免继续滑向结构化补丁。

## 4. 产品需求

### 4.1 回复质量需求

1. 正常客户问题必须由 LLM 作为“大脑”理解和组织回复。
2. 回复必须直接回应客户当前问题，不能被旧上下文带偏。
3. 对明确问题要给明确建议，不能模棱两可。
4. 对闲聊、问候、告别，要自然回应，有人情味，不机械转卖车。
5. 对无关但无害的话题，可以短暂陪聊，再柔性引导回业务。
6. 对复杂回复要拆成 1-3 条短句，每条都要有完整含义。
7. 所有客户可见回复必须由 Brain 发出，并经过最终润色校验；不能因提速跳过 Brain 或润色。
8. 回复要像真人客服，不暴露 AI、系统、规则、知识库、模型、API 等内部信息。

### 4.2 事实权威需求

1. 商品库是商品事实最高权威。
2. 正式知识库是政策、流程、风控边界最高权威。
3. 当前会话事实只在当前会话内有效。
4. AI经验池、真实聊天、历史数据只作为风格和经验参考，不能作为事实依据。
5. LLM常识层可以做泛化分析，但不能生成商品事实或业务承诺。
6. 候选知识不能直接参与客户可见回复。
7. 如果商品库没有对应商品，不得编造价格、库存、公里数、车况。
8. 如果正式知识没有明确政策，不得承诺贷款、过户、优惠、售后等确定结果。

### 4.3 多会话需求

1. 每个会话有独立上下文、独立消息批次、独立 LLM 任务、独立待发送队列。
2. 任何回复发送前必须重新确认目标会话正确。
3. 多个会话同时产生 ready reply 时，只允许串行发送，按 ready 时间和安全优先级排序。
4. 不允许把 A 会话的回复发到 B 会话。
5. 不允许把 OCR 识别出来的会话名、用户名当成正文交给 LLM。

### 4.4 速度需求

1. 第一优先级是不降低回复质量。
2. 第二优先级是减少程序内尾延迟。
3. 不允许用本地固定模板替代 LLM 润色来提速。
4. 允许优化证据包大小、Prompt 缓存、并发 LLM、ready 唤醒、发送前复核耗时。
5. 上游 LLM 不稳定不计入本程序缺陷，但程序必须具备 failover 和清晰错误报告。

### 4.5 RPA需求

1. 本次重构不改变 RPA 操作微信的底层策略。
2. RPA 层只负责捕获、切换、输入、发送、异常检测，不负责业务回复判断。
3. 发送前仍需保持严格目标复核。
4. 不因 Brain First 引入更频繁、更机械的微信前台操作。

## 5. 非目标

本次设计不做：

- 不重写 RPA 控制层。
- 不重做云端授权。
- 不重做商品库后台。
- 不重做 AI记录员结构化导出。
- 不删除现有规则。
- 不把 LLM 放开到可以编造事实。
- 不取消最终润色。
- 不取消人工转接接口。

## 6. 目标架构

```text
RPA捕获
  -> 会话归属与正文清洗
  -> 硬安全预检
  -> 权威证据包构建
  -> LLM客服大脑
  -> 事实与边界Guard
  -> 最终可见润色
  -> 发送前目标复核
  -> RPA串行发送
```

## 7. 分层职责

### 7.1 RPA捕获层

职责：

- 识别未读会话。
- 捕获当前会话消息。
- 清理 OCR 噪音。
- 保留原始截图、原始 OCR 和清洗后正文。

不得做：

- 不判断业务意图。
- 不生成客服回复。
- 不因某条消息短而默认忽略。

### 7.2 会话归属与正文清洗层

职责：

- 将 OCR 中的发言人名、会话名、时间、系统提示剥离成元数据。
- 为客服模块和记录员模块提供统一清洗后的正文。
- 生成稳定的消息 ID 和去重键。

关键规则：

- `许聪\n在吗` 应转成 `speaker=许聪`、`content=在吗`。
- 群聊中 `张三: 报价多少` 应转成 `speaker=张三`、`content=报价多少`。
- `文件传输助手`、`新数据测试` 等会话名不得当成客户正文。

### 7.3 硬安全预检层

职责：

- 识别不可回复或必须转人工的场景。
- 识别微信掉线、白屏、RPA焦点异常、窗口错位等执行风险。
- 阻止 prompt injection、内部信息探测、违法违规要求。

不得做：

- 不处理正常销售回复。
- 不因为“信息少”就跳过客户消息。
- 不用固定模板覆盖普通会话。

### 7.4 权威证据包层

职责：

- 从商品库、正式知识库、当前会话、AI经验池、常识层、风格层构建证据包。
- 明确每条 evidence 的权威等级和可用范围。
- 明确哪些证据可作为内容依据，哪些只能作为风格参考。

权威顺序：

```text
商品库
正式知识库
当前会话事实
LLM常识层
话术风格层
AI经验池
候选知识
```

注意：权威顺序不是“都可当事实”。客户可见事实依据只允许商品库、正式知识库、当前会话事实。

### 7.5 LLM 客服大脑层

职责：

- 理解客户当前问题。
- 结合上下文判断客户真实意图。
- 做别名、错别字、音译、指代消解。
- 判断应直接回答、追问、柔性陪聊、转人工还是引用商品/正式知识。
- 输出回复策略和 1-3 条候选短句。

不得做：

- 不编造商品事实。
- 不编造政策承诺。
- 不绕过 guard。
- 不暴露内部系统。

### 7.6 Guard层

职责：

- 校验 LLM 输出的事实是否有权威依据。
- 校验是否越过正式知识边界。
- 校验是否答非所问。
- 校验是否过长、是否拆句完整。
- 校验是否需要转人工。

### 7.7 最终可见润色层

职责：

- 让回复更自然、口语、人味、短句化。
- 保留 Brain 的策略和事实。
- 不新增事实。
- 不改变风险动作。

### 7.8 发送层

职责：

- 串行发送。
- 发送前目标复核。
- 发送中窗口有效性检查。
- 失败后停止并报告，不盲目重复。

## 8. 旧模块的新定位

| 模块 | 旧定位 | 新定位 |
| --- | --- | --- |
| `realtime_reply_router.py` | 可直接生成本地回复 | 路由和风险分级器，正常回复不得抢 Brain 决策 |
| `rag_answer_layer.py` / RAG reply | 可生成回复并跳过 LLM | 证据提供者，不直接定稿 |
| `llm_reply_synthesis.py` | 后置综合回复 | Brain First 的核心实现基础 |
| `reply_evidence_builder.py` | synthesis 的证据包 | Brain 输入证据包构建器 |
| `llm_reply_guard.py` | synthesis 守卫 | Brain 输出事实与边界守卫 |
| `final_visible_llm_polish.py` | 最终润色 | 保留，但只做表达层优化 |
| style memory | 风格参考 | 保留，只影响表达 |
| AI经验池 | 经验治理 | 保留，不作为事实依据 |

## 9. 新旧流程对比

### 9.1 旧流程

```text
客户消息
  -> 本地规则 / RAG / realtime route 尝试回答
  -> 如果 route 允许，再调用 LLM synthesis
  -> 最终润色
```

问题：

- LLM 不一定参与思考。
- 本地 route 可能生成机械答案。
- RAG 命中可能跳过 LLM。
- 真实客户长尾表达需要不断补规则。

### 9.2 新流程

```text
客户消息
  -> 只做硬安全和证据准备
  -> 默认进入 LLM Brain
  -> Guard 验证事实和边界
  -> 最终润色
```

优势：

- LLM 默认参与理解。
- 真实客户长尾表达由大模型处理。
- 商品事实仍被商品库约束。
- 政策边界仍被正式知识和 guard 约束。
- 本地规则不再抢答。

## 10. 验收原则

重构后必须满足：

1. 正常业务问题默认进入 Brain。
2. RAG 命中不能跳过 Brain。
3. realtime route 不能在正常业务场景直接生成最终回复。
4. 所有回复仍经过 final visible polish。
5. 商品报价必须来自商品库。
6. 政策规则必须来自正式知识库。
7. 闲聊、问候、告别自然，不机械转业务。
8. 多会话不串上下文、不串回复。
9. 实盘测试不出现白屏、掉线、错发。
10. 速度不能明显劣化，程序内尾延迟要有分项观测。
