# 权威分层 RAG 与 AI经验池总体方案

## 目标

本方案把微信智能客服的知识流转统一收束为“入口治理、出口受控”的架构：

1. 信息入口统一先进入 AI经验池，商品库专用入口除外。
2. AI经验池负责收集、理解、审计、分发，不直接参与客户可见回答决策。
3. 需要进入正式知识的内容，必须先进入候选知识层，再由人工确认升级。
4. 客户回复的内容依据只允许来自商品库、正式知识库、当前会话事实。
5. LLM常识层只做通用分析，话术风格层只做表达润色，均不得生成或覆盖事实。
6. 最大化复用现有 `rag_experience`、`review_candidates`、`style_memory`、`product_master` 和正式知识库代码，不推倒重做。

## 命名调整

现有代码中的 `rag_experience` 在数据结构上继续保留，产品和文档语义调整为：

- 技术名：`rag_experience`
- 产品名：`AI经验池`
- 新定位：原始经验治理中枢

原因：

- 现有 `RagExperienceStore`、治理统计、后台列表、导入链路已经可用。
- 直接改目录和表名风险高，迁移成本大。
- 可以先保留技术名，在 UI 和文档中逐步改称“AI经验池”。

## 核心原则

### 入口原则

所有非商品库专用入口的数据先进入 AI经验池：

- 历史聊天记录。
- 微信实时聊天中新产生的数据。
- 上传的普通资料。
- AI自动记录员抽取出的资料。
- 人工手动输入的经验。
- 客服修正、失败案例、转人工案例。
- 运行时产生的回复质量观察。

商品库数据例外：

- 通过商品库专用导入、商品库编辑、商品主数据同步进入的数据，可以直接进入 `product_master`。
- 如果商品事实来自聊天、普通上传、记录员文本，则只能进入商品候选，不得直接覆盖商品库。

### 出口原则

客户回复时，内容依据只允许来自：

1. 商品库。
2. 正式知识库。
3. 当前客户会话事实。

辅助层只能辅助：

- AI经验池：治理、审计、分发、候选生成，不直接参与回答内容依据。
- 话术风格层：只影响表达方式，不影响事实和规则。
- LLM常识层：只做通用取舍分析，不生成商品事实或业务承诺。
- 候选知识层：待审，不参与客户可见回答。

## 新层级定义

| 层级 | 职责 | 是否可作为客户回复内容依据 |
| --- | --- | --- |
| 商品库 | 商品事实唯一最高权威源，包括价格、库存、车型、配置、里程、车况、是否在售 | 是 |
| 正式知识库 | 稳定业务规则、流程、政策、风控边界、转人工规则 | 是 |
| 当前会话事实 | 当前客户刚表达的预算、用途、联系方式、预约时间、偏好 | 是，仅限当前会话 |
| LLM常识层 | 通用行业常识、非事实型取舍分析、表达组织 | 辅助，不可造事实 |
| 话术风格层 | 从真实销售话术中学习表达风格、节奏、自然度 | 只润色，不决策 |
| 候选知识层 | 待人工确认的正式知识或商品更新线索 | 否 |
| AI经验池 | 原始数据汇总、LLM深度分析、路由分发、审计留痕 | 否 |

## AI经验池的新职责

AI经验池不再被理解为“运行时可检索答案库”，而是以下工作流的中枢：

1. **收集**
   - 接收所有非商品库专用入口的信息。
   - 保留来源、时间、会话、操作者、租户、原始文本、附件摘要。

2. **理解**
   - 使用 LLM 对信息做结构化分析。
   - 输出事实线索、规则线索、话术模式、客户意图、风险标签、时效性判断。

3. **分发**
   - 话术模式自动写入话术风格层。
   - 稳定业务规则生成正式知识候选。
   - 商品事实线索生成商品更新候选。
   - 转人工或风控案例生成转人工规则候选。
   - 无效、过期、低质内容自动归档或建议废弃。

4. **审计**
   - 每次分发都有来源、目标层、理由、风险、操作者、时间、版本。
   - 人工可否决、删除、回滚。

5. **治理**
   - 周期性去重、降噪、冲突检查。
   - 对过期或高风险经验自动降级。

## 分发目标

| 输入类型 | 默认目标 | 是否自动生效 |
| --- | --- | --- |
| 纯话术表达、客服语气、邀约方式 | `style_memory` | 可自动生效，但必须事实脱敏 |
| 稳定流程、门店政策、风控边界 | `review_candidates` | 否，人工确认后进入正式知识 |
| 商品事实线索 | `product_master` 候选 | 否，人工确认或商品库专用入口后生效 |
| 转人工场景 | 转人工规则候选 | 否，人工确认后生效 |
| 常识型沟通方法 | LLM常识提示候选或正式知识候选 | 否 |
| 噪音、重复、过期、一次性内容 | 归档或废弃 | 可自动归档，删除需留审计 |

## 运行时回答链路

新的运行时回答链路：

```text
用户消息
  -> 当前会话事实抽取
  -> 商品库检索
  -> 正式知识库检索
  -> LLM常识层做受限分析
  -> 生成候选回复
  -> 话术风格层润色
  -> 事实/规则守卫
  -> 输出给用户
```

明确禁止：

```text
AI经验池 / 历史聊天 / 普通上传资料 / 候选知识
  -> 直接作为客户回复内容依据
```

允许：

```text
AI经验池
  -> 分发到 style_memory / review_candidates / product candidates
  -> 经人工或专用入口确认后
  -> 商品库 / 正式知识库
  -> 运行时依据
```

## 当前差距

当前系统已经有大部分基础设施，但存在几个不符合新原则的地方：

1. 部分真实聊天经验仍可进入 `rag_index` 并作为 `rag_evidence`。
2. `AI经验池` 的 UI 文案容易让人误解为可直接参与回答。
3. `reply_evidence_builder` 仍保留 `rag_experience` / `rag_evidence` 作为运行时知识字段。
4. 话术风格层还需要更强的事实脱敏，避免历史价格、车型、承诺被原句带入。
5. 自动分发还不是统一的“经验池 -> 分发计划 -> 目标层”机制。
6. 候选知识、商品候选、风格记忆之间的自动分发审计还不够统一。

## 新硬约束

### 约束一：内容依据白名单

客户可见回复的内容依据只允许：

- `product_master`
- `formal_knowledge`
- `product_scoped_formal`
- `current_conversation_fact`

### 约束二：聊天数据黑名单

以下来源不得作为内容依据：

- `cleaned_real_chat_pack`
- `real_chat_style`
- `wechat_raw_message`
- `raw_wechat_private`
- `raw_wechat_group`
- `raw_wechat_file_transfer`
- `ai_recorder_chat`
- `rag_experience`
- `review_candidate`

这些来源可以用于：

- 话术风格。
- 意图理解辅助。
- 候选生成。
- 审计复盘。
- 转人工风险提示。

### 约束三：事实脱敏

写入话术风格层前，应移除或模板化：

- 具体价格。
- 库存数量。
- 车架号、手机号、姓名。
- 具体车型和唯一商品描述。
- 里程、年份、上牌时间等商品事实。
- 绝对承诺，如“包过”“保证无事故”“一定能办”。

### 约束四：候选默认不生效

所有候选知识默认不可参与客户回复。只有人工确认升级后，才进入正式知识库。

### 约束五：旧索引必须清理

迁移后必须保证：

- `rag_index` 中没有聊天来源经验。
- `rag_evidence` 不再把 AI经验池内容传给 LLM 作为内容依据。
- chejin 现有真实聊天样本全部降级为风格参考或候选，不再可检索为内容证据。

## 验收口径

完成后必须能证明：

1. 历史聊天不会作为内容依据进入客户回复。
2. 商品事实只来自商品库。
3. 业务规则只来自正式知识库。
4. 当前客户刚说的信息可以用于当前会话，但不会沉淀为事实知识直接复用。
5. 话术风格层可以影响表达，但不会带入历史事实。
6. AI经验池可以自动生成分发计划，但不能直接影响客户回复内容。
7. 所有分发都有审计记录、人工否决和回滚路径。
