# 微信RPA客服质量优先上下文与尾延迟优化方案（2026-06-03）

## 1. 本轮结论先行

本轮不采纳以下“提速捷径”：

1. 短问候如“你好 / 在吗 / 再见”直接走本地固定模板。
2. 明确报价如“秦PLUS多少钱”直接用本地模板发出。
3. 短回复跳过最终润色。

原因：这些路径虽然能降低平均耗时，但会让回复更机械、更重复，也会破坏“真人客服感”。当前第一优先级仍然是：**回复拟人化、自然、不答非所问、不机械重复**。速度优化只能发生在不损伤这个目标的环节。

## 2. 质量边界

### 2.1 必须保留

1. 主回复生成仍走现有业务理解链路。
2. 最终可见回复仍经过 `final_visible_llm_polish`。
3. 商品报价、车况、配置等事实依据仍以商品库为最高优先级。
4. 正式知识库仍作为政策、流程、售后、贷款置换等稳定知识依据。
5. AI经验池、话术风格层、常识层只辅助表达与理解，不覆盖商品库/正式知识库事实。

### 2.2 可以优化

1. RPA OCR 识别出的“发言人名字”不得混入正文参与思考。
2. 会话内“这辆 / 这台 / 刚才那款 / 报价”等相对追问必须绑定上一轮明确商品上下文。
3. 多会话任务必须专人专用，回复生成完成后发送前再次校验目标会话，避免错发。
4. 程序内安全探针、轮询等待、ready reply 唤醒等尾延迟可以优化。
5. 最终润色的调度方式可以并发和复用，但不能关闭、不能降质。

## 3. 当前暴露的问题

### 3.1 OCR正文污染

RPA OCR 会把类似：

```text
许聪
在不在
```

识别成同一条正文。旧逻辑在部分私聊/未知会话里不会剥离第一行“许聪”，导致：

1. 客服模块把“许聪”当成客户正文一起交给 LLM。
2. 记录员模块可能把“在不在”误抽成姓名字段。
3. 对话理解漂移，回复出现无依据的“还差电话”等内容。

### 3.2 商品上下文断裂

用户先问“比亚迪秦PLUS车况”，随后问“这辆什么价格”时，程序应当理解“这辆”指向上一轮秦PLUS。如果会话状态没有稳定记录上一款车，就可能退回泛化推荐，甚至报出宝马/奥迪等无关车型。

### 3.3 程序尾延迟

现有数据里，部分耗时来自上游 LLM，但仍有本地可控长尾：

1. 被动安全探针可能挡在新消息处理前。
2. ready reply 已经润色完成后，到真正发送之间存在二十秒级尾巴。
3. 多会话捕获时如果一轮只处理一个真实未读，会话二会排队等待。

### 3.4 多会话错发风险

后台 LLM 可以并发，但微信前台只能串行。任何 ready reply 发送前都必须重新确认：

1. 目标会话名一致。
2. 当前窗口非白屏、非掉线。
3. 输入框干净。
4. 即将发送的回复属于当前 target/session。

## 4. 优化方案

### 4.1 发言人前缀清洗

新增/补强统一规则：

1. 如果 OCR 正文第一行是已知会话名、已知发言人名、或本轮目标配置中的联系人名，且第二行存在有效正文，则第一行只作为 `speaker_name` 元数据。
2. 清洗后正文进入客服回复、记录员抽取、历史锚点、去重键和 LLM。
3. 原始 OCR 文本保留到 `original_content`，方便审计。
4. 群聊和私聊都走同一套清洗入口，避免客服与记录员行为不一致。

### 4.2 会话商品上下文稳定化

新增/补强会话上下文字段：

1. `last_product_id`
2. `last_product_name`
3. `last_product_price`
4. `last_product_source`
5. `last_product_updated_at`

当用户明确提到某个商品，或系统证据包最终绑定到某个商品时，写入会话状态。后续出现“这辆 / 这台 / 刚才那个 / 报价 / 车况”等相对追问时，优先把商品库中该商品注入证据包，再交给 LLM 生成和润色。

注意：这不是本地模板直发。它只是把“事实依据”锁定正确，最终表达仍由 LLM 和最终润色完成。

### 4.3 质量优先的报价路径

报价类问题的目标路径：

```text
用户问价
  -> 商品库/上下文锁定商品事实
  -> 业务回复生成
  -> 最终可见润色
  -> 发送前会话复核
  -> RPA发送
```

因此，“秦PLUS多少钱”如果商品库有价格，回复必须包含正确报价，但不走机械模板，不跳过润色。

### 4.4 尾延迟优化

本轮只优化程序内等待，不牺牲回复质量：

1. 安全探针不应挡在已有未读/ready reply 之前。
2. ready reply 形成后要主动唤醒发送循环。
3. 空闲时保持低扰动；发现未读时立即进入工作态，只在切会话前保留 1-3 秒随机人类化延迟。
4. 保持微信前台发送串行，避免错发和风控。

### 4.5 测试策略

必须覆盖：

1. `许聪\n在不在` 清洗后正文为 `在不在`，发言人为 `许聪`。
2. `新数据测试` 私聊/未知会话也能清洗其他已知联系人名。
3. 客服和记录员都不把发言人名当正文。
4. 先问秦PLUS车况，再问“这辆什么价格”，回复必须围绕秦PLUS且包含商品库价格。
5. 短问候仍走 LLM/最终润色，不走固定模板。
6. 多会话 ready reply 发送前仍严格复核目标。
7. 调度测试验证安全探针不会阻塞已有未读和 ready reply。

## 5. 验收标准

1. 静态检查通过。
2. 工作流测试、调度测试、OCR兼容测试通过。
3. 二手车专项回复测试通过，尤其是商品追问和报价绑定。
4. 低风险实盘测试不出现白屏、掉线、错发、草稿残留。
5. 质量路径确认：短回复和报价回复仍经过最终可见润色。
