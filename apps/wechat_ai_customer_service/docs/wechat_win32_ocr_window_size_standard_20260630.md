# WeChat Windows OCR 窗口尺寸标准

## 背景

Windows OCR sidecar 启动交互动作前会把微信主窗口归一化到一个可识别、可点击、可截图的尺寸。
这个尺寸必须根据当前机器的屏幕坐标系计算，不能简单按 DPI 放大。

问题根因是：部分 Windows API 返回的是后续 `MoveWindow` 实际使用的逻辑坐标，而不是显示器标称物理
像素。比如一台 1920 宽、125% 缩放的机器，在程序里可能表现为 1536 宽逻辑屏。如果再把默认窗口
`980x860` 乘以 1.25，就会变成 `1225` 宽，视觉上接近占据大半个屏幕。

## 标准规则

启动前必须读取当前屏幕尺寸，并使用这个尺寸作为窗口规划输入：

- `screen_width`
- `screen_height`
- `dpi_scale` 仅用于诊断记录，不参与默认窗口尺寸放大

窗口默认目标尺寸按当前 Windows 逻辑屏幕档位决定：

| 屏幕逻辑分辨率 | 推荐目标窗口 |
| --- | --- |
| 小于 `2400x1350` | `980x860` |
| 大于等于 `2400x1350` 且小于 `3200x1800` | `1225x1075` |
| 大于等于 `3200x1800` | `1470x1290` |

`1920x1080`、`1920x1200`，以及它们在 125%/150% 缩放下对应的逻辑分辨率，都属于第一档，必须保持
旧默认窗口 `980x860`。不能因为 DPI 是 1.25 或 1.5 就放大窗口。

## 可见区域裁剪

如果屏幕放不下推荐目标窗口，只裁剪最终执行尺寸，不改变推荐目标的语义：

- 宽度最多使用 `screen_width - 12`
- 高度最多使用 `screen_height - 48`
- 极小屏幕下可以进一步裁到真实屏幕尺寸

示例：

| 当前逻辑屏 | DPI | 推荐目标 | 实际执行尺寸 |
| --- | --- | --- | --- |
| `1920x1080` | `1.0` | `980x860` | `980x860` |
| `1536x864` | `1.25` | `980x860` | `980x816` |
| `1536x960` | `1.25` | `980x860` | `980x860` |
| `1280x720` | `1.5` | `980x860` | `980x672` |
| `2560x1440` | `1.0` | `1225x1075` | `1225x1075` |
| `3840x2160` | `1.5` | `1470x1290` | `1470x1290` |

## 环境变量覆盖

`WECHAT_WIN32_OCR_WINDOW_WIDTH` 和 `WECHAT_WIN32_OCR_WINDOW_HEIGHT` 仍是显式覆盖入口。

默认情况下 `WECHAT_WIN32_OCR_ENFORCE_RECOMMENDED_WINDOW=1`，如果覆盖值小于当前档位推荐目标，会被提升
到推荐目标；如果覆盖值更大，会被安全上限和屏幕可见区域裁剪。

## 验收要求

- 1920 档及其高缩放逻辑屏不会放大到 `1225` 或 `1470`。
- 当前屏幕分辨率必须出现在 `window_normalization.screen` 元数据中。
- `dpi_scale` 可以记录，但不能决定默认窗口目标尺寸。
- 小屏只裁剪，不强行撑大到半屏或满屏。
