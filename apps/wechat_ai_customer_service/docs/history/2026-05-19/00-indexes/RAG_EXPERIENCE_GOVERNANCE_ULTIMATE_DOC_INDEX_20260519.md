# RAG 经验治理终极改造文档索引（2026-05-19）

## 1. 目标

本组文档用于在正式代码落地前，统一规划“RAG经验层 -> 候选知识 -> 正式知识库 -> 运行时回复”的终极治理形态。

本轮不是只修“很可靠/建议废弃”显示冲突，而是从底层解决：

- RAG经验状态自相矛盾；
- 自动吸纳与AI建议互相覆盖不清；
- 清洗实盘聊天到底进入经验层、风格层还是候选层不清；
- 高价值经验没有自动提名候选；
- 商品主数据、正式知识、RAG经验、话术风格样本边界容易再次混淆。

## 2. 推荐阅读顺序

1. `RAG_EXPERIENCE_GOVERNANCE_ULTIMATE_ARCHITECTURE_20260519.md`
   解释终极架构、层级职责、状态机、最终裁决优先级。

2. `RAG_EXPERIENCE_GOVERNANCE_ULTIMATE_DATA_CONTRACT_20260519.md`
   定义新增字段、状态枚举、兼容规则和不变量。

3. `RAG_EXPERIENCE_GOVERNANCE_ULTIMATE_IMPLEMENTATION_GUIDE_20260519.md`
   按阶段列出代码落地点、修改顺序和每阶段验收标准。

4. `RAG_EXPERIENCE_GOVERNANCE_ULTIMATE_MIGRATION_RUNBOOK_20260519.md`
   说明 chejin 现有冲突数据如何备份、dry-run、修复、回滚。

5. `RAG_EXPERIENCE_GOVERNANCE_ULTIMATE_TEST_ACCEPTANCE_20260519.md`
   定义静态测试、单元测试、数据回归、前端验收和实盘验收清单。

## 3. 终极目标

最终系统应满足：

- 所有自动学习入口先进入 RAG 经验层或风格层，不直接写正式知识。
- RAG经验有统一的“最终治理裁决”，不再出现同一条同时“自动吸纳”和“建议废弃”。
- 高价值、低风险、可泛化的 RAG经验可自动生成“待确认知识候选”，但不能自动入正式库。
- 商品主数据只允许商品库人工导入或维护，不能从 RAG经验、候选知识或风格层反向写入。
- 清洗实盘聊天可以学习真人表达，但商品事实、动态推荐、边界承诺不得作为权威事实。
- 前端只展示一个用户能理解的最终处理结果，同时保留质量评分、AI建议和来源审计作为辅助信息。
- 总数统计严谨：总经验数必须等于所有显式状态数量之和，不能隐藏未知状态。

## 4. 非破坏性原则

本轮改造必须保持以下边界：

- 不改动正式知识“人工审核后入库”的安全底线。
- 不恢复任何 AI 自动写正式知识库的路径。
- 不允许商品主数据从经验链路晋升。
- 不允许文件传输助手、自测 marker、AI自回复进入学习池。
- 不把 `quality.band=high` 等同于“可用知识”。
- 不让旧的系统自动状态永久压过新的硬规则。
- 不静默覆盖用户人工处理结果。

## 5. 阶段总览

| 阶段 | 名称 | 目标 | 是否改变运行时回复 |
| --- | --- | --- | --- |
| Phase 0 | 基线冻结与审计 | 备份、统计冲突、确认当前数据状态 | 否 |
| Phase 1 | 最终治理裁决层 | 新增统一 resolver，消除状态冲突 | 仅影响RAG经验是否可检索 |
| Phase 2 | 前端统一展示 | 用户只看到一个最终处理结论 | 否 |
| Phase 3 | 历史数据修复 | 修复 chejin 现有 auto_kept/discard 冲突 | 影响错误经验检索准入 |
| Phase 4 | 自动候选提名 | 高价值经验自动生成 pending candidate | 不自动入库 |
| Phase 5 | 风格层协同 | 实盘聊天可 style_only，不误当事实 | 影响表达风格，不授权事实 |
| Phase 6 | 全量测试与实盘回归 | 验证架构、数据、前端、回复质量 | 是，做验收 |

## 6. 开工纪律

- 先写测试再改核心状态机。
- 每个阶段只能改本阶段清单内的文件。
- 每个阶段通过后才能进入下一阶段。
- 任何数据修复必须先 dry-run 和备份。
- 任何自动候选提名必须只写 pending candidate，不能应用到正式库。
- 前端文案必须面向普通业务用户，不能暴露内部状态名作为主说明。
