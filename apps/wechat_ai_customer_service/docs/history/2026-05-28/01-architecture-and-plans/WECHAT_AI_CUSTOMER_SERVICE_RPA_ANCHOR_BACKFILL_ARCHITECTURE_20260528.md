# 微信自动客服 RPA 锚点追溯架构方案（2026-05-28）

## 1. 背景

现有自动客服已经有消息去重和历史补读能力，但当前实盘风险主要来自两点：

1. 冷启动或状态不完整时，旧可见消息可能被当成新消息处理。
2. 历史补读以固定次数上翻为主，不能做到“找到边界立即停止”。

这两个问题的本质不是历史回滚本身，而是缺少稳定的“回复边界”。没有边界时，系统只能靠未处理 ID、内容指纹和当前可见窗口猜测哪些消息是新消息。

## 2. 设计原则

1. 回滚是必要能力，不一刀切禁用。
2. 回滚必须有明确目标：寻找上一轮已确认处理锚点。
3. 当前屏幕能定位锚点时，不允许启动历史回滚。
4. 找到锚点后立即停止，不继续机械上翻。
5. 找不到锚点时宁可暂停或转人工，不盲目回复。
6. 回滚动作应低扰动、有限速、有限步数、可审计。
7. RPA 操作仍保持串行，LLM 思考可以并发。

## 3. 核心概念

### 3.1 会话锚点

会话锚点是“系统确信此处之前已处理、此处之后才可能有新消息”的边界。

优先级如下：

1. 上一轮成功发送回复所覆盖的客户消息 ID。
2. 上一轮成功发送回复所覆盖的客户消息内容指纹。
3. 上一轮自己发出的回复文本指纹。
4. 最近若干条已处理客户消息的内容片段指纹。
5. 最近一次 bootstrap 基线事件中的可见消息指纹。

### 3.2 新消息区间

找到锚点后，锚点之后到当前最新位置之间的客户文本消息就是候选新消息区间。

系统只把这个区间交给批次选择、语义规划、LLM 回复和发送前复检。

### 3.3 缺口风险

如果当前可见窗口找不到锚点，且底部存在未处理消息，说明中间可能有消息缺口。此时不能只看最后几条就回复，应进入锚点追溯。

## 4. 目标流程

```text
切到目标会话
  -> 读取当前可见消息，不滚动
  -> 构造会话锚点集合
  -> 在当前可见窗口定位最新锚点
       -> 找到：提取锚点之后的新消息区间
       -> 未找到：判断是否需要追溯
            -> 不需要：跳过或仅记录
            -> 需要：小步上翻，截图识别，查找锚点
                 -> 找到：停止上翻，合并锚点之后的新消息区间
                 -> 未找到且达到上限：暂停或转人工
  -> 批次选择与语义规划
  -> LLM 思考
  -> 发送前再次按同一机制复检
  -> 发送成功后更新锚点
```

## 5. 与当前机制的差异

| 项目 | 当前机制 | 新机制 |
|---|---|---|
| 回滚触发 | 未处理数量、窗口饱和、锚点缺失等粗条件 | 当前屏幕找不到上一轮处理锚点且存在缺口风险 |
| 回滚停止 | 固定 `load_times` | 找到锚点立即停止 |
| 旧新分界 | 已处理 ID 和内容 key 去重 | 上一轮成功回复边界优先，去重作为辅助 |
| 找不到边界 | 可能继续基于可见消息判断 | 暂停或转人工，不盲发 |
| 审计能力 | 记录是否补读、补读次数 | 记录锚点来源、搜索步数、停止原因和消息区间 |

## 6. 锚点选择策略

每个会话保存一个 `last_successful_reply_anchor`，它来自成功发送后的 `mark_processed`。

该锚点应包含：

- 本轮客户消息 ID 列表。
- 本轮客户消息内容指纹列表。
- 本轮发送出去的回复文本指纹。
- 回复时间。
- 发送校验结果。
- 关联的 `reply_trace_id`。

查找时按“越新越可靠”的原则排序。若同屏出现多个锚点，使用最新位置作为边界。

## 7. 追溯策略

追溯不是一次滚很多，而是循环执行：

1. 小幅上翻。
2. 短暂停顿。
3. 截图 OCR。
4. 合并当前快照。
5. 查找锚点。
6. 若找到则停止。

每次追溯都受以下限制：

- `max_scroll_steps`
- `max_duration_seconds`
- `max_snapshots`
- `max_messages_after_load`
- `min_delay_ms` / `max_delay_ms`
- 单会话冷却时间

该策略的目标是降低误操作和异常高频 UI 行为，而不是绕过或对抗平台安全机制。

## 8. 发送前复检

发送前复检必须复用同一套锚点追溯逻辑。

如果 LLM 思考期间客户又发了新消息：

- 当前回复对应的原批次仍在锚点之后，且没有更新消息：可以发送。
- 出现更新消息：当前回复标记为 stale，重新合并新消息后再思考。
- 原批次和锚点都找不到：暂停或转人工。

## 9. 冷启动与 bootstrap

冷启动不能把历史消息直接当新消息回复。

推荐策略：

1. 启动前读取当前可见窗口，不上翻。
2. 将当前可见文本消息标为 bootstrap 基线。
3. 等待之后的新消息触发回复。
4. 如果用户明确要求“补看历史”，进入人工授权的历史恢复模式。

若需要启动时补全上下文，也应使用锚点追溯，而不是无边界固定次数回滚。

## 10. 失败处理

以下情况必须暂停或转人工：

1. 达到最大追溯步数仍找不到锚点。
2. OCR 质量过低，无法稳定识别消息顺序。
3. 目标会话识别不确定。
4. 检测到微信登录异常、白屏、辅助壳窗口、消息区域阻塞。
5. 发送前复检发现回复已过期且无法重建完整上下文。

## 11. 结论

最合理的机制不是禁用历史回滚，也不是继续固定次数补读，而是把历史回滚改成“锚点搜索”。这样既能保证消息完整性，又能减少不必要的 UI 操作，并且能明确说明每次回复到底基于哪一段消息。
