# 微信自动客服 RPA 锚点追溯风险登记（2026-05-28）

## 1. 主要风险

| 风险 | 影响 | 控制措施 |
|---|---|---|
| OCR 把同一条消息识别成不同文本 | 锚点找不到或重复处理 | 保存完整 key、归一化 key、长文本片段 key |
| 客户连续发送相同内容 | 内容 key 误判边界 | 优先 ID 和最新位置，内容 key 只作辅助 |
| 锚点离开可追溯范围 | 无法确定完整上下文 | 达到上限后暂停或转人工 |
| 回滚动作过多 | 影响稳定性，增加异常 UI 行为 | 小步、限次、限时、找到即停 |
| 追溯后没有回到最新 | 后续发送位置错误 | `restore_to_latest=true`，发送前再确认输入框 |
| 多会话并发下状态串扰 | 错回或串回 | 每会话独立锚点，RPA 操作串行 |
| 冷启动旧消息被当新消息 | 重复回复 | 当前可见 bootstrap 基线，不自动历史回滚 |
| 找不到锚点仍回复 | 业务事故 | `block_on_anchor_not_found=true` |

## 2. 不可做事项

以下行为不进入本方案：

- 伪造硬件、设备、网络、登录环境。
- 绕过微信登录、风控或安全机制。
- 多 RPA worker 同时操作同一个微信窗口。
- 无边界无限上翻。
- 实盘默认固定次数历史回滚。
- 找不到锚点时让 LLM “猜一下上下文”。

## 3. 降级策略

| 状态 | 降级动作 |
|---|---|
| 锚点找不到 | 暂停自动回复，创建转人工事件 |
| OCR 质量低 | 暂停自动回复，保存截图和审计 |
| 微信窗口异常 | 停止 runtime，推送告警 |
| 发送前 stale | 丢弃旧回复，重新入队 |
| 多次追溯失败 | 暂时禁用该会话自动回复 |

## 4. 告警事件

建议统一导向现有转人工接口：

- `anchor_not_found_gap_risk`
- `anchor_search_ocr_low_confidence`
- `anchor_search_target_not_confirmed`
- `stale_reply_blocked`
- `wechat_login_or_shell_detected`
- `wechat_window_blocked`

## 5. 商用前要求

商用前必须确认：

1. 所有追溯行为都有审计记录。
2. 所有阻断行为都能推送到飞书或 ServerChan。
3. 客户可见回复不会暴露“机器人、AI、自动化、转人工内部机制”等内部表述。
4. 运维界面能看见“暂停原因”和“最近一次锚点搜索结果”。
5. 可以一键禁用自动追溯，退回只读当前窗口的保守模式。

## 6. 最终判断

锚点追溯机制的验收标准不是“永远不滚动”，而是“只在必要时滚动，滚动有目标，找到即停，找不到不回”。这比禁用回滚更完整，也比固定次数回滚更安全。

## 7. 后续自测待办清单

本清单记录 2026-05-28 锚点追溯实盘验收后的后续自测项，优先用于后续迭代排期。

| 优先级 | 自测项 | 目标 |
|---|---|---|
| P0 | 找不到锚点阻断测试 | 人为构造 state 中存在旧锚点但当前历史不可达，验证必须暂停或转人工，不能强行回复 |
| P0 | 多轮连续锚点测试 | 连续 3 轮“发消息-回复-再发消息”，验证每轮都以最近一次成功回复边界为新锚点 |
| P0 | 重复/相似消息测试 | 多条内容高度相似、同一句拆开发送时，不误把错误消息当锚点，也不重复处理 |
| P1 | OCR 换行鲁棒性测试 | 不同窗口宽度导致长消息换行不同，`message_content_key` 仍能匹配 |
| P1 | 小窗口/窗口归一化测试 | 微信窗口偏小、拖动或非目标会话时，能恢复到安全尺寸并确认目标后再操作 |
| P1 | 低频长测 | 30-60 分钟低频实盘，观察白屏、踢下线、重复回复、残留进程和误滚动 |

## 8. 微信 RPA 高风险行为与优化方向

以下内容只用于减少异常 UI 行为、降低误触和稳定性风险；不得用于伪造设备、绕过登录、安全验证或逆向规避平台安全策略。

### P0：必须避免或硬阻断

| 高风险行为 | 风险表现 | 优化方向 |
|---|---|---|
| 多个 RPA worker 同时操作微信 | 串会话、误发、拖拽、白屏 | 全局 RPA lock；LLM 并发但 RPA 读写串行 |
| 未确认目标会话就发送 | 发错人、发群、发到测试外联系人 | 发送前 OCR 标题确认；目标不确认直接停机/转人工 |
| 登录页、二维码页、白屏后继续自动修复和发送 | 可能扩大异常状态，增加账号风险 | 识别到登录/白屏/invalid hwnd 后立即停机并告警 |
| 找不到历史锚点仍自动回复 | 漏读上下文、重复或错回 | `block_on_anchor_not_found=true`，转人工 |
| 高频连续发送或多会话压力刷屏 | 发送行为过密、窗口频繁切换 | 发送限频、burst 上限、低频灰度、冷却窗口 |
| 无边界上翻或固定次数机械回滚 | 重复 wheel/OCR 行为异常，且可能拖动/误选文字 | 锚点驱动、小步限次、找到即停、找不到阻断 |
| 使用剪贴板一次性大段粘贴 | 行为突兀，也容易污染用户剪贴板 | 默认 sendinput_unicode 分段输入；仅测试刷屏时临时允许复制粘贴 |
| 在用户手动操作时抢鼠标键盘 | 误拖、误点、输入错位 | operator guard 锁键鼠；F8 暂停/停止；悬浮状态同步 |

### P1：需要节流、降级或审计

| 高风险行为 | 风险表现 | 优化方向 |
|---|---|---|
| 频繁切换会话列表 | 会话列表 OCR 抖动，微信渲染压力增加 | 优先 active/pending 会话；白名单低频轮扫；每轮目标数上限 |
| 频繁 passive/interactive 探针 | 前台焦点反复变化，窗口恢复过度 | passive 探针低频；interactive 只在启动/人工触发时执行 |
| 发送失败后重复重试 | 客户看到重复内容，发送频次异常 | unknown/send_failed 不自动重发；记录并转人工 |
| 输入框清空动作过重 | 误全选、误删、误选聊天区文字 | 输入区 OCR 确认；仅在输入区有草稿时清理；不对聊天区做全选 |
| 滚动后不恢复到底部 | 后续发送/读取位置错乱 | `restore_to_latest=true`；发送前再次确认输入框与目标标题 |
| LLM 思考时间过长但旧回复继续发送 | stale 回复 | 发送前 freshness check；有新消息则丢弃旧回复重算 |
| OCR 质量低仍继续判断 | 锚点误判、漏消息 | OCR 低置信/消息数异常时暂停，并保存截图审计 |

### P2：体验与运营优化

| 行为 | 风险表现 | 优化方向 |
|---|---|---|
| 回复节奏过于固定 | 客户体验像模板 | LLM 思考时间自然分布；发送前轻微随机延迟；但不为了“像人”增加无意义动作 |
| 每轮都触发完整知识/LLM链路 | 响应慢，队列积压 | 常规问题走快速路由；高风险才转人工或更深推理 |
| 调试脚本留下测试状态 | 正式目标被替换、误跑测试会话 | 实盘脚本必须备份/恢复 settings/config/state，并有后置验收 |

## 9. 优先优化队列

1. P0 找不到锚点阻断专项测试。
2. P0 多轮连续锚点测试。
3. P0 重复/相似消息锚点误匹配测试。
4. P1 OCR 换行和小窗口归一化测试。
5. P1 低频 30-60 分钟长测。
6. P1 发送失败 unknown 状态的转人工与去重验证。

## 10. 红线

- 不做设备指纹、硬件地址、网络身份伪装。
- 不绕过微信登录、扫码、安全校验或风控流程。
- 不逆向、篡改微信客户端安全逻辑。
- 不为了“模拟真人”制造无意义的大量误触、拖动、反复切换或重复消息。
- 出现登录页、白屏、invalid hwnd、目标不确认、输入框不可用、锚点 gap risk 时，先停机，再告警，再人工确认。

## 11. 沙盒双号测试复盘与下轮注意事项

### 11.1 本轮容易触发风险的因素

| 类型 | 现象 | 结论 |
|---|---|---|
| 配置错误 | `respond_all_unread_sessions=true` 或设置中存在额外启用目标，导致运行时扩展到 `新数据测试昨天19:23` 等非本轮目标 | 这是明确流程错误，必须通过启动硬门禁阻断 |
| 目标错位 | 沙盒/宿主窗口切换后，当前会话不一定处于可发送状态，必须点击联系人名称或确认标题后才能发送 | 必须把“目标标题确认”作为发送硬前置 |
| 旧回滚机制 | 固定回滚或不带锚点的 wheel/OCR 在错误会话上触发，容易造成机械滚动和窗口异常 | 已改为锚点驱动；下轮仍需验证“不必要不回滚” |
| 异常后继续恢复 | 白屏、blank/loading、invalid hwnd 后继续做窗口恢复、探针或发送，会扩大异常操作 | 必须 hard stop，转人工/告警后等待人工确认 |
| 双端都被 RPA 操作 | 沙盒端发消息、宿主端读取/回复、再切回沙盒确认，短时间内频繁前台切换 | 下轮只允许一次只操作一个端；宿主 AI 客服自动回复时，不并行操作沙盒 |
| 消息模式过于测试化 | 纯数字时间戳、连续相似短消息、固定间隔，测试味太重 | 使用自然、低频、内容不同的真实咨询话术 |
| 沙盒环境天然变量 | 新安装、新登录、虚拟化窗口、同机双号、窗口渲染偶发 blank/loading | 只能降低测试强度和增加冷却，不做设备/网络伪装 |

### 11.2 下轮沙盒真人模拟前置门禁

必须全部满足才允许开始：

1. 宿主微信和沙盒微信窗口不重叠，宿主固定左侧，沙盒固定右侧。
2. 两端都已稳定登录，窗口无白屏、无登录页、无 blank/loading。
3. 宿主 AI 客服设置只启用本轮目标 `许聪`，`respond_all_unread_sessions=false`。
4. 沙盒发送目标只允许 `Meta_xc` 和 `文件传输助手`，禁止触达其他联系人或群。
5. RPA passive acceptance 通过；wxauto4 未启用；无残留 listener/operator guard/recorder/live 进程。
6. 发送前 OCR 标题确认、输入区确认、目标可发送状态确认均通过。
7. 若任一确认失败，不自动修复多次，不继续发送，直接停机并告警。

### 11.3 下轮测试节奏

建议按以下灰度推进：

| 阶段 | 动作 | 通过标准 |
|---|---|---|
| A | 沙盒手动或低频 RPA 给 `Meta_xc` 发 1 条自然问题 | 宿主正确收到，无白屏/掉线 |
| B | 宿主 AI 客服回复 1 条 | 目标确认正确，回复发送成功，沙盒可见 |
| C | 冷却 2-3 分钟后再发第 2 条 | 无重复回复、无 stale 回复、无异常探针 |
| D | 连续 3 轮低频问答 | 每轮锚点更新正确，RPA 操作次数可解释 |
| E | 再做小规模边界：长消息、换行、补充消息 | 找不到锚点时阻断，找到锚点时正常回复 |

### 11.4 测试内容规则

- 使用自然咨询，不使用连续纯数字、连续时间戳或高度重复话术。
- 每条消息内容不同，像真实客户逐步补充需求。
- 不做群发、不做多会话压力、不做快速来回切换。
- 沙盒端发送后等待宿主处理完成，不同时操作宿主和沙盒。
- 不使用剪贴板批量刷屏，除非单独声明为刷屏边界测试并降低频率。

### 11.5 停止条件

出现以下任一情况立即停止：

- 任一账号被踢下线、出现登录页或扫码页。
- 微信白屏、blank/loading、invalid window handle。
- 当前会话标题不是目标。
- 输入框不可确认或聊天区被误选中。
- 发送结果 unknown 或 send_failed。
- 自动客服扩展到非白名单会话。
- 锚点 gap risk 无法恢复。
