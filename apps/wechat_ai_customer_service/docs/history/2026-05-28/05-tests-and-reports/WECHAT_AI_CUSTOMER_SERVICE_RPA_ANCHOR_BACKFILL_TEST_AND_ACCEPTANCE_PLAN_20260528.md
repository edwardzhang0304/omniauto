# 微信自动客服 RPA 锚点追溯测试与验收计划（2026-05-28）

## 1. 测试目标

验证新机制能同时满足：

- 不漏读：客户刷屏或窗口被顶出时能追溯到边界。
- 不重复：旧消息不会被当成新消息再次回复。
- 不乱回：找不到边界时不盲发。
- 少扰动：没有必要时不回滚，必要时有限步追溯。
- 可复盘：每次追溯都有明确审计字段。

## 3. 单元场景

| 场景 | 预期 |
|---|---|
| 当前屏幕有上一轮客户消息锚点 | 不调用历史追溯，锚点之后消息被选中 |
| 当前屏幕有上一轮回复文本锚点 | 不调用历史追溯，锚点之后消息被选中 |
| 当前屏幕无锚点但无新消息 | 跳过，不追溯 |
| 当前屏幕无锚点且有未处理消息 | 调用锚点追溯 |
| 追溯后找到锚点 | 停止追溯，合并锚点之后消息 |
| 追溯后仍找不到锚点 | 返回 blocked，不发送 |
| OCR 文本分行变化 | 内容片段 key 仍能命中 |
| 重复内容消息 | 不能只靠内容误判，需结合 sender/type/位置 |

## 4. 发送前复检场景

| 场景 | 预期 |
|---|---|
| LLM 思考期间无新消息 | 正常发送 |
| LLM 思考期间客户补充一条 | 原回复 stale，重新合并思考 |
| 原批次不在屏内但追溯找到 | 基于锚点后消息判断是否 stale |
| 原批次和锚点都找不到 | 阻断发送，转人工 |

## 5. 模拟集成场景

使用 fake connector 构造多窗口消息：

1. 10 条连续客户消息，锚点在当前屏内。
2. 30 条刷屏，锚点在第 2 次上翻后出现。
3. 50 条刷屏，达到上限仍找不到锚点。
4. 客户消息和客服回复交错，确保只收客户消息。
5. 多会话调度下一个会话追溯，不阻塞其他会话 LLM 任务。

## 6. 受控实盘测试

实盘测试应低频、短链路、逐步扩大：

1. 单会话 1 条新消息，确认不回滚。
2. 单会话 3 条连续补充，确认合并回复。
3. 当前屏幕内存在上一轮锚点，确认不回滚。
4. 人工制造轻度刷屏，让上一轮锚点离开当前屏幕，确认小步追溯并找到即停。
5. 发送前补充新消息，确认旧回复不发送。
6. 找不到锚点的安全场景，确认暂停或转人工。

每轮实盘都必须记录：

- 目标会话。
- 发出的客户测试文本。
- 是否触发追溯。
- 追溯步数。
- 停止原因。
- 是否发送。
- 是否出现白屏、踢下线、辅助壳窗口或窗口错位。

## 7. 验收标准

通过条件：

1. 当前屏幕有锚点时，历史追溯调用次数为 0。
2. 锚点缺失时，追溯在找到锚点后立即停止。
3. 找不到锚点时不会自动回复。
4. 旧消息不会因为冷启动或 OCR ID 改变而重复回复。
5. 发送前复检能拦截 LLM 思考期间的新消息。
6. 审计字段完整可读。
7. 全量静态和逻辑测试通过。
8. 实盘测试无白屏、无踢下线、无错会话发送。

## 8. 验收红线

出现以下任一情况不得交付：

- 自动客服向非目标会话发送。
- 找不到锚点仍继续发送。
- 当前屏幕有锚点仍机械回滚。
- 追溯达到上限后没有暂停或转人工。
- 发送前发现新消息仍发送旧回复。
- 实盘中触发微信登录异常但没有停机告警。
