# 微信自动客服实时响应与 token 成本优化文档索引（2026-05-16）

## 1. 本轮目标

本轮改造目标是把微信自动客服从“每条消息都可能同步等待大模型综合回复”的模式，调整为“实时响应稳定优先，LLM 只在必要时参与，慢任务全部后台化”的模式。

核心要求：

- 回复速度尽量快，常规消息不再被重型 LLM 调用拖住。
- token 消耗显著下降，避免每条实盘消息都消耗上万 tokens。
- 不牺牲回复质量，尤其不能降低二手车场景里的真实客服口吻、商品事实准确性和风险边界。
- 任何外部模型、微信适配器、文件锁或网络波动都不能让监听主循环长时间停住。
- 保持“防暴露 AI 身份”策略，边界问题使用请示/核实话术，对内继续告警和接管。

## 3. 与旧文档的关系

2026-05-07 版前后台分离文档中有一条旧约束：每次回复必须经 LLM 合成。该约束在本轮被替换为：

> 每次回复必须经过统一回复裁决与安全 guard；LLM 合成只在证据不足、表达质量不足、复杂上下文或高价值场景中参与，且不能无限阻塞实时回复链路。

这不是降低智能程度，而是把智能分成两类：

- 在线智能：小证据包、强时限、可兜底，服务客户实时体验。
- 离线智能：深度分析、经验沉淀、候选知识、话术优化，服务长期质量提升。

## 4. 预期收益

基于 2026-05-16 实盘慢回复审计记录，当前一次普通预算推荐可能消耗约 12k-14k tokens，历史记录中还出现过 intent advisory 与 reply synthesis 双调用，单条消息可能达到 20k+ tokens。

目标上线后：

- 常规问候、留资、边界请示、明确商品事实：前台 0 LLM tokens。
- 常规商品推荐：优先本地筛选 + 轻量话术，必要时小包 Flash，约 600-1800 tokens。
- 复杂多轮上下文：小包 Flash，约 1500-3000 tokens。
- 高风险/强承诺：默认请示话术 + 告警，前台 0 LLM tokens；后台可异步 Pro 分析。
- 离线学习、知识治理、经验解释：继续允许 Pro，但不阻塞客户回复。

综合预计：

- 实盘前台 token 消耗降低 70%-95%。
- 普通消息平均响应从偶发数分钟降低到 2-8 秒内。
- 即使模型或网络异常，主监听循环也应在硬超时后恢复，不再整段卡死。

## 5. 修改前必须确认的约束

- 商品资料仍然是最高确定性主数据，只能由商品库入口维护，不能由 RAG 经验或聊天学习反向写入。
- RAG 经验只提供风格、表达、相似问法和软性经验，不能授权价格、库存、合同、金融、车况承诺。
- 防暴露 AI 身份开启时，客户可见话术不能出现“转人工/AI/机器人”等暴露机制的表达。
- 所有后台 LLM 结果都必须可审计，不允许静默改正式知识。
- 实时链路任何外部调用都必须有单步 timeout 和整轮 watchdog。
