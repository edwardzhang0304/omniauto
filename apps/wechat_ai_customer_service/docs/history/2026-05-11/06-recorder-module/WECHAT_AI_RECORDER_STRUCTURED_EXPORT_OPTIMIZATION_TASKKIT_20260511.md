# 微信AI智能记录员结构化导出优化任务包（2026-05-11）

## 1. 目标与范围

本轮目标：在不破坏 OmniAuto 通用底座的前提下，优化“结构化导出（实验仪器订货表V1）”能力，提升可用性、稳定性与可运维性。

范围分两层：

1. 通用底座层（所有客户共用）：
- 导出任务体验一致（按钮逻辑、状态提示、自动轮询）。
- 日期筛选能力通用化（按日/按周/按月/自定义范围）。
- 默认导出上限统一为 `10000`。

2. 客户模块层（仅 `order_sheet_lab_v1`）：
- 规则 + LLM 混合抽取质量优化。
- 日期字段标准化（优先输出 `YYYY-MM-DD`）。
- 降低 LLM 调用耗时与 token 消耗风险。

## 2. 差距与修复清单

### 2.1 通用底座差距

1. 前端部分文案仍残留旧术语“创建结构化导出任务”。
2. 空状态说明与按钮文案不一致，用户认知成本高。
3. 测试脚本仍断言旧文案，存在回归误判风险。

修复动作：

1. 全量统一前端文案为“导出所有记录（结构化）”。
2. 去除无效刷新按钮引用，保持自动轮询为唯一状态刷新机制。
3. 同步修订回归测试断言文本。

### 2.2 模块层差距（order_sheet_lab_v1）

1. 仍存在日期格式不统一（可能出现 `MMDD`、`M月D日` 混杂）。
2. 多行消息在“单行先命中”时可能跳过更优的多行组合抽取。
3. 低质量行（仅金额无货品）可能被误导出。
4. LLM 补全触发范围偏宽，长任务耗时与 token 成本偏高。
5. 模块默认配置仍偏旧（`date_output_mode=MMDD_text`、`llm_max_rows_per_run=40`）。

修复动作：

1. 增加日期归一化：统一解析并按模块配置输出（默认 `YYYY-MM-DD`）。
2. 调整抽取策略：优先保留“有货品名”的行，缺货品时回退多行组合推断。
3. 增加噪音/取消语义过滤，抑制非订货语句进入导出。
4. 收敛 LLM 补全触发条件，仅在关键字段缺失时触发。
5. 下调默认 LLM 预算（降低耗时与成本）。

## 3. 设计原则（模块可切换）

为满足“admin 账号按客户切换能力模块”的目标，约束如下：

1. 通用导出任务框架不绑定任何具体行业规则。
2. 行业逻辑仅在模块分支内生效（如 `order_sheet_lab_v1`）。
3. 模块配置（日期格式、LLM预算、内容类型）可被 VPS Admin/本地模块注册表覆盖。
4. 新客户只需新增模块并切换绑定，不修改底座流程。

## 5. 验收标准

1. UI：
- 主按钮显示“导出所有记录（结构化）”。
- 高级筛选可直接触发“按日期范围导出”。
- 导出任务状态自动更新，无需手工刷新按钮。

2. 导出结果：
- 日期字段默认输出 `YYYY-MM-DD`。
- 低质量噪音行明显下降。
- 订货关键信息（货品、数量、金额）提取完整度提升。

3. 成本与性能：
- 默认 LLM 调用预算低于旧版。
- 同等数据量下平均等待时长下降或不劣于旧版。

4. 架构：
- 通用底座与客户模块仍可解耦切换。

## 6. 风险与回滚

风险：

1. 规则收紧导致召回下降（漏提取）。
2. 规则放宽导致噪音上升（误提取）。
3. LLM预算过低影响复杂消息补全。

回滚策略：

1. 模块配置回滚：恢复旧 `date_output_mode`、`llm_max_rows_per_run`。
2. 代码回滚：仅回退 `order_sheet_lab_v1` 分支逻辑，不影响通用导出框架。
3. 绑定回滚：VPS Admin 将目标账号切回 `raw_message_log_v1` 保障可用性。
