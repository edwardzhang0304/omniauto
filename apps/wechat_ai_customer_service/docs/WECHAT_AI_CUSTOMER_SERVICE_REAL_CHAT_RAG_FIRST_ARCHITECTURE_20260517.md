# 实盘聊天 RAG-First 架构修复方案

## 背景问题

chejin 二手车账号中，清洗后的真实客服聊天样本曾通过标准 workflow import 被直接写入 `knowledge_bases/chats/items`。这导致正式知识库中出现大量 `chejin_real_*` 条目。它们虽然有学习价值，但本质是历史对话样本，不是稳定、通用、可审计的正式规则。

直接写入正式知识库会带来三个问题：

1. 权威层混乱：正式知识库应该回答“应该怎么说”，实盘聊天样本只是“过去怎么说过”。
2. 质量不可控：真实聊天含有客户场景、一次性表达、边界问题、旧规则，不能默认升级为正式话术。
3. 加载与治理变慢：数百上千条样本进入正式库后，前端分页、编译、检索都会被拖慢。

## 修复后的分层

### 1. 商品主数据层

职责：保存确定的商品/车源事实，例如车型、价格、库存、里程、配置、车况摘要。

允许来源：人工维护、资料导入中的商品入口、授权的商品主数据导入脚本。

禁止来源：RAG经验、聊天记录、清洗后的实盘聊天、候选知识晋升。

### 2. 正式知识库层

职责：保存稳定、通用、可审计的业务规则和正式话术模板。

允许来源：人工创建、审核后的候选知识、明确声明为正式模板的导入文件。

禁止来源：未经人工改写的实盘聊天样本、微信原始记录、清洗后的客服对话包。

### 3. RAG经验层

职责：保存可复用经验、上传资料摘要、真实聊天样本、AI回复后的效果经验。

允许来源：资料上传、微信原始聊天、清洗后的实盘聊天包、运行时RAG回复经验。

晋升规则：高价值经验可以进入候选知识；候选知识必须人工审核后才能写正式知识库。商品资料例外：永远不能从 RAG 晋升。

### 4. 实盘话术风格层

职责：保存真实客服的语气、问法承接、追问节奏、边界缓冲方式。

允许来源：清洗后的实盘聊天、人工挑选的话术样本、RAG中已确认的风格样本。

禁止职责：不承担事实权威，不输出车源价格库存，不作为商品事实来源。

## Real Chat 硬边界

凡满足任一条件，都属于 real_chat 类来源：

- `source.type` 为 `cleaned_real_chat_pack`、`real_chat`、`wechat_raw_message`、`raw_wechat_private`、`raw_wechat_group`、`raw_wechat_file_transfer`。
- 文件或条目 id 命中 `chejin_real_*` 等实盘聊天批次命名。
- `source.batch_token` 包含 `realchat`、`real_chat`、`实盘聊天` 等来源语义。

real_chat 类来源的默认落点只能是：

- RAG经验层。
- 话术风格记忆层。
- 人工审核候选池。

real_chat 类来源的禁止落点：

- 正式知识库直接写入。
- 商品主数据写入。
- 自动候选直接确认入库。

## 导入入口统一原则

“资料导入”可以继续作为用户唯一入口，但后台必须按来源自动分流：

- 商品资料：进入商品主数据层。
- 规则/政策资料：进入 RAG经验，必要时生成候选等待人工审核。
- 聊天记录/清洗实盘对话：进入 RAG经验和话术风格层。
- 明确正式模板：只有带有正式模板声明且不属于 real_chat 时，才允许 workflow apply 写正式库。

## 为什么这不是降低学习能力

修复后，实盘聊天仍会被学习，只是不再错误地变成权威知识。程序会通过两条路径使用它：

- RAG经验层：帮助理解类似客户问法和历史处理方式。
- 话术风格层：帮助回复更接近真人客服的表达。

正式知识库则保持干净，用于承载稳定规则。这种分层能同时保证“像真人”和“不会乱编事实”。
