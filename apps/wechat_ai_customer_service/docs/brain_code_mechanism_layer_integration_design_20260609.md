# Brain层与代码机制层分层融入总体方案（2026-06-09）

## 客户可见回复所有权硬基线

本方案继承并强制引用 [customer_visible_reply_ownership_baseline.md](customer_visible_reply_ownership_baseline.md)：

- 所有客户可见回复必须由 `customer_service_brain` 发出。
- Guard、质量门、语义审稿、RAG、实时路由、本地模板、旧合成器、最终润色和任何兜底模块不能生成、替换、拼接客户可见回复。
- Brain 不可用、超时、不可采纳或返修失败时，必须阻断发送、记录审计并触发内部人工/告警接口，不能发送本地 fallback。

## 1. 设计目标

过去一段时间的优化已经建立了商品库、正式知识库、AI经验池、话术风格、LLM常识、Brain First、RPA账本优先等能力。但实盘暴露出两个空档：

1. **实时思考缺少独立层级名称。**
   很多“如何理解客户、如何处理闲聊、如何在不越界时给明确建议、如何结合上下文继续回答”的原则散落在 Brain prompt、contract、质量门、旧路由和测试里。它们本质上属于运行时客服大脑，应明确命名为 `Brain层`。

2. **代码执行正确性缺少独立层级名称。**
   会话识别、session_key、会话账本、OCR 辅助、未读识别、RPA动作、发送前核验、白屏/掉线检测、防风控动作节奏等规则不是知识，也不是话术。它们属于程序运行机制，应明确命名为 `代码机制层`。

本方案的目标是把这两层补进现有体系，让所有规则都有归属，避免继续出现“临时补丁散落代码、guard越权、旧模板复活、RPA调度无总纲”的问题。

## 2. 完整分层体系

| 层级 | 核心职责 | 是否可授权客户可见事实 | 是否可生成客户可见回复 |
| --- | --- | --- | --- |
| 商品库层 | 商品名称、别名、价格、库存、年款、里程、车况、位置、配置、是否在售 | 是，商品事实最高权威 | 否 |
| 正式知识层 | 稳定政策、流程、合同、金融、置换、售后、风险边界、转人工规则 | 是，政策流程最高权威 | 否 |
| 当前会话事实层 | 客户本会话刚表达的预算、用途、偏好、联系方式、预约时间、上下文指代 | 是，仅限当前会话 | 否 |
| 共享公共策略层 | 跨账号通用客服原则、实体归一、自然表达、闲聊处理、缺信息追问方式 | 否，只能辅助 Brain 策略 | 否 |
| LLM常识层 | 通用常识、行业取舍分析、非事实型建议、表达组织 | 否，不能编造商品或政策事实 | 否 |
| 话术风格层 | 真实聊天中沉淀的表达节奏、语气、短句习惯、销售风格 | 否，只能影响表达风格 | 否 |
| AI经验池 | 原始资料入口、LLM理解、治理、审计、候选分发、风格提炼 | 否，不直接参与运行时事实依据 | 否 |
| 候选知识层 | 待人工确认的正式知识、商品更新、共享公共知识候选 | 否，确认前不生效 | 否 |
| Brain层 | 实时理解客户、选择证据、处理上下文、做回复策略、生成 BrainPlan | 只能引用上游授权事实 | 是，唯一客户可见回复作者 |
| Guard/质量/润色审稿层 | 审核 Brain 输出的安全、事实、完整性、自然度，给返修意见 | 否 | 否 |
| 代码机制层 | OCR/RPA/调度/session/账本/发送核验/异常保护/性能预算/防机械动作 | 否 | 否 |

### 2.1 会话策略状态的归属

连续闲聊、套话、身份试探、抗拒业务牵引等多轮节奏问题，统一纳入 [conversation_strategy_state_design_20260609.md](conversation_strategy_state_design_20260609.md)。

该机制不是新的事实层，也不是新的回复出口：

- `代码机制层` 按 `session_key` 维护 `conversation_strategy_state`，只记录节奏、疲劳度和策略提示。
- `共享公共策略层` 定义“连续无业务意图后弱化业务牵引”的通用原则。
- `Brain层` 消费该状态并决定本轮怎么自然回应。
- Guard/质量/润色层只检查是否过度硬拉业务、是否越界、是否暴露内部机制，并把问题反馈给 Brain。
- 商品库、正式知识和当前会话事实的权威边界不变。

## 3. 新增 Brain层定义

### 3.1 Brain层负责什么

Brain层是运行时客服回复的唯一决策中心，负责：

- 理解客户当前消息，包括短问候、追问、反问、质疑、闲聊、找茬、错别字、同音/音译/简称。
- 读取商品库、正式知识、当前会话事实和共享策略。
- 使用 AI经验池、历史聊天、话术风格、LLM常识作为辅助材料，但不能把它们当成事实授权。
- 决定是否直接回答、追问、给推荐、比较、安抚、拒绝、边界说明或请求人工介入。
- 生成 `BrainPlan.reply_segments`，每条都是可独立发送的完整微信短句。
- 接收 Guard/质量门/语义审稿意见后重新思考和返修。

### 3.2 Brain层不能做什么

Brain层不能：

- 编造商品库不存在的价格、库存、车况、配置、优惠。
- 编造正式知识不存在的政策、流程、合同、赔付、金融承诺。
- 用历史聊天、AI经验池或风格记忆中的旧事实回答当前客户。
- 绕过 guard、质量门、最终可见校验。
- 把 `fallback_existing`、旧模板或本地路由结果当作上级答案。

### 3.3 Brain层输入合同

Brain 每次运行应拿到分层标注过的输入：

- `current_message`：本轮客户消息，已去除 OCR speaker label。
- `session_context`：当前会话事实、近期上下文、最近已发回复。
- `product_master_evidence`：商品库候选与事实字段。
- `formal_knowledge_evidence`：正式知识、商品专属正式知识。
- `shared_strategy_guidance`：共享公共策略、实体归一、表达原则。
- `style_guidance`：风格层摘要，不含事实授权。
- `common_sense_guidance`：常识层建议，不含事实授权。
- `review_feedback`：guard/quality/reviewer 的返修意见。
- `mechanism_metadata`：session_key、capture_id、message_digest 等仅用于归属与调度，不作为话术内容。

### 3.4 Brain层输出合同

Brain 输出仍以 `BrainPlan` 为准：

- `answer_mode`
- `reply_strategy`
- `evidence_used`
- `facts_claimed`
- `reply_segments`
- `risk`
- `recommended_action`
- `handoff_reason`
- `reason`

新增要求：

- `reply_segments` 是唯一客户可见正文来源。
- `facts_claimed` 只能引用商品库、正式知识、当前会话事实。
- `common_sense_topics`、`style_notes`、`shared_strategy_used` 只能放辅助说明，不得成为事实来源。
- `mechanism_metadata` 不得进入客户可见文本。

## 4. 新增代码机制层定义

### 4.1 代码机制层负责什么

代码机制层是系统运行正确性的底座，负责：

- 微信窗口发现、激活、置顶、尺寸/遮挡/白屏/掉线检测。
- OCR 消息识别、speaker label 分离、头像未读红点/预览/可见消息识别。
- `session_key` 生成、会话账本、消息 digest、context_version、capture_id。
- 多会话调度、并发 LLM 任务、ready reply 队列、先来后到发送。
- 发送前核验：当前会话、session_key、capture、账本、message digest、freshness。
- RPA动作策略：点击区域随机化、输入节奏、发送动作、避免重复点击、避免同时键鼠异常行为。
- 键鼠锁、悬浮球、F8暂停/停止、运行状态同步。
- 异常停机：微信掉线、白屏、渲染异常、窗口失效、风控风险、发送目标不一致。
- 性能预算：轮询节奏、低扰动监听、LLM 并发队列、超时策略。

### 4.2 代码机制层不能做什么

代码机制层不能：

- 写客户可见回复。
- 根据关键词决定业务回复。
- 用本地模板兜底发送客户消息。
- 改变 Brain 的业务策略。
- 把 OCR 识别到的用户名、群名、会话标题当作客户正文。
- 在无法确认会话归属时继续发送。

### 4.3 代码机制层核心数据合同

每个 capture、Brain task、ready reply、send 操作必须携带：

- `session_key`
- `target_name`
- `conversation_type`
- `capture_id`
- `message_ids`
- `input_content_keys`
- `message_content_digest`
- `context_version`
- `last_visible_anchor`
- `reply_id`
- `capture_source`
- `ledger_state_version`
- `send_target_confirmed`

这些字段只用于系统归属和安全核验，不允许进入客户可见回复。

## 5. 与 AI经验池/RAG 机制的关系

新增 Brain层和代码机制层不会和 AI经验池冲突，原因如下：

| 维度 | AI经验池 | Brain层 | 代码机制层 |
| --- | --- | --- | --- |
| 时间属性 | 异步沉淀、治理、候选分发 | 实时回复决策 | 实时系统执行 |
| 处理对象 | 原始资料、聊天、失败案例、记录员数据 | 当前客户问题和证据包 | 微信窗口、OCR、RPA、队列、账本 |
| 是否回答客户 | 否 | 是，唯一出口 | 否 |
| 是否授权事实 | 否 | 否，只引用授权事实 | 否 |
| 是否能改商品/正式知识 | 只能生成候选 | 不能 | 不能 |

RAG 仍然是技术手段，不是独立权力层：

- 对商品库、正式知识做向量/语义检索，可以成为 Brain 的权威证据。
- 对 AI经验池、历史聊天做向量/语义检索，只能成为风格、经验、风险提示，不得授权商品或政策事实。
- RAG 不能绕过 Brain 直接出客户可见回复。

## 6. 运行时链路

```text
微信新消息
  -> 代码机制层识别未读/预览/OCR
  -> 代码机制层生成 capture，写入 session ledger
  -> 证据构建层按权威顺序取商品库、正式知识、当前会话事实、共享策略、风格和常识
  -> Brain层生成 BrainPlan.reply_segments
  -> Guard/质量/润色审稿层只审核和反馈
  -> 若需返修，审稿意见回到 Brain层
  -> 代码机制层执行发送前核验
  -> RPA串行发送
  -> 代码机制层写入 reply_sent 和审计
```

## 7. 学习沉淀链路

```text
导入资料/聊天记录/记录员抽取/运行观察
  -> AI经验池
  -> LLM理解和分发计划
  -> 话术风格层 / 候选知识层 / 商品候选 / 共享公共知识候选 / 废弃归档
  -> 人工确认后进入正式知识或商品库
  -> 运行时由 Brain 读取授权层
```

## 8. 分层归属原则

新增或修改任何规则前，必须先回答：

1. 这是事实吗？
   - 商品事实进商品库。
   - 政策流程进正式知识。
   - 当前客户刚说的信息进当前会话事实。

2. 这是通用客服思路吗？
   - 进共享公共策略层或 Brain层 prompt 合同。

3. 这是风格吗？
   - 进话术风格层，且必须脱敏。

4. 这是原始经验或失败案例吗？
   - 进 AI经验池，由它分发候选。

5. 这是运行正确性吗？
   - 进代码机制层。

6. 这是审稿意见吗？
   - 进 Guard/质量/润色审稿层，只能反馈给 Brain。

7. 这是为了某个案例临时加的话术吗？
   - 默认拒绝。应转成通用 Brain 策略、数据修复、证据修复或测试用例。

## 9. 当前应收拢的重点

优先把以下散落内容收拢：

1. `customer_service_brain.py` 与 `customer_service_brain_contract.py` 中的通用回复策略。
   - 保留为 Brain层合同。
   - 能数据化的迁入共享公共策略层。
   - 具体产品词不得留在策略代码里。

2. `listen_and_reply.py`、`realtime_reply_router.py`、`llm_reply_synthesis.py` 中的旧可见模板。
   - Brain First 模式下必须隐藏或降级为 advisory。
   - 不允许作为客户可见出口。

3. `llm_reply_guard.py`、`final_visible_llm_polish.py` 中的审稿规则。
   - 硬边界进入正式知识或平台安全规则。
   - 软质量问题只反馈给 Brain。

4. RPA、多会话、账本、发送前确认、低扰动监听。
   - 统一归为代码机制层。
   - 文档、代码和测试中使用同一术语。

5. 共享公共知识缓存中的旧式“无证据即转人工”规则。
   - 应改为：没有授权事实时，Brain 可以说明需要核实、自然追问或请求人工，但软缺证据不能直接压过 Brain。

## 10. 验收标准

完成分层收拢后，应满足：

- 任意客户可见回复都能追溯到 `BrainPlan.reply_segments`。
- 任意商品事实都能追溯到商品库或商品专属正式知识。
- 任意政策流程都能追溯到正式知识。
- 任意历史聊天只能影响风格或候选，不影响事实。
- 任意 RPA/调度/发送逻辑都归属于代码机制层。
- Guard、质量门、润色层没有客户可见回复所有权。
- 多会话并发时，reply envelope 与 session ledger 能防止错发。
- 短问候、闲聊、常识问题仍由 Brain 自然回复，不被 guard 误伤。
