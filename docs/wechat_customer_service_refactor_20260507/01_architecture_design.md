# 微信AI客服前后台分离架构设计

## 版本
- 设计日期：2026-05-07
- 状态：待实施
- 关联任务：前台极速回复 + 后台慢速分析改造

## 背景与约束

### 用户核心诉求
1. **前台**：最快速度给出客户回答，尽量精准且不越界
2. **后台**：慢慢分析，把该入库的、该吸收的经验逐步处理好，慢没关系
3. **硬性约束**：LLM合成不能跳过，必须保持拟人化，避免机械化表达

### 当前架构问题
- 所有工作都在 `listen_and_reply.py --once` 同步串行执行
- 每次回复延迟约 5-10 秒（意图识别 1-2s + 回复合成 2-5s + 后置工作 0.5-1s + overhead）
- 无独立后台 worker，经验吸收、数据入库、深度分析均阻塞前台回复路径

## 目标架构

### 前台（Foreground）—— 目标延迟 2.5-4s

前台只保留"与客户回复直接相关"的工作，其他全部移除或异步化。

```
读消息 (~0.5s)
  │
  ├─ 并行启动 ─┬─ 意图识别 (flash, 2s timeout) ──┐
  │            ├─ 商品知识匹配                    │
  │            ├─ RAG 检索                        │
  │            └─ 数据字段提取                    │
  │                                               ▼
  │                                        回复合成 (flash为主, pro兜底)
  │                                               │
  │                                        Guard 质检
  │                                               │
  └──────────────────────────────────────── 发送消息 (~0.5s)
                                               │
                                               ▼
                                        立即返回，前台结束
                                        （后置工作：Enqueue 后台任务）
```

**前台必须保留的阻塞步骤（不可省）：**
- 意图识别：决定走哪条分支，必须用 LLM 保证精准
- 回复合成：负责拟人化表达，必须保留
- Guard 质检：安全边界检查，不可跳过
- 消息发送：最终动作

**前台可异步化/移除的步骤：**
- 客户数据写入 Excel
- 原始消息记录与归档
- state 文件全量持久化（可改为增量/延迟）
- RAG 经验吸收与审核
- 知识库编译
- 对话摘要与深度分析

### 后台（Background）—— 无时效要求

利用已有 `WorkQueueService`（`admin_backend/services/work_queue.py`）作为任务队列基础设施。

后台 Worker 进程 `background_worker.py` 独立运行，从队列中 `claim()` 任务并执行：

| 任务类型 | 说明 | 模型要求 | 优先级 |
|---------|------|---------|--------|
| `experience_interpretation` | RAG 经验深度解读、业务类型标注 | pro（慢没关系） | 5 |
| `rag_quality_audit` | 新吸收经验的质量审核、风险检查 | pro | 5 |
| `knowledge_compile` | 知识库编译到磁盘 | 无 LLM | 3 |
| `conversation_summary` | 长对话摘要、客户画像更新 | pro | 6 |
| `customer_data_sync` | 客户数据批量写入/同步 Excel | 无 LLM | 4 |
| `diagnostics_deep_check` | 全量语义诊断、LLM 知识审计 | pro | 7 |
| `raw_message_archive` | 原始消息归档、去重清理 | 无 LLM | 8 |

## 模型分层策略（不变）

| 任务 | 模型 | 说明 |
|------|------|------|
| 意图识别 | **flash** | 2s timeout，快速分类 |
| 回复合成（普通） | **flash** | 默认路径，90% 查询 |
| 回复合成（高风险） | **pro** | 金融/合同/售后/转人工 |
| 后台经验解读 | **pro** | 慢没关系，追求质量 |

## 关键原则

1. **拟人化不可妥协**：每次回复必须经 LLM 合成，system prompt 已强化 `natural_reply_style` 规则
2. **前台极简**：前台进程只做"读→识→合→发"，其余一律 enqueue
3. **后台充分利用 WorkQueueService**：已有的队列基础设施（文件/Postgres 双后端、去重、重试、锁）直接复用
4. **渐进改造**：分 P0/P1/P2 三层实施，每层可独立验证
